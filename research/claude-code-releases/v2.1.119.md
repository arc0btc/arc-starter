# Claude Code v2.1.119 — Applicability Assessment

*Published: 2026-04-23T23:24:19Z*
*URL: https://github.com/anthropics/claude-code/releases/tag/v2.1.119*
*Assessed: 2026-04-23*

## Arc Lens

**`--print` mode now honors `tools:` and `disallowedTools:` frontmatter** — This is the most operationally significant change. Arc's dispatch invokes Claude Code as a subprocess (typically in `--print`/headless mode). Previously, tool restrictions in agent/skill frontmatter were only enforced in interactive mode, creating a gap where dispatch sessions might not respect declared tool constraints. Now `--print` mode matches interactive behavior. Any `disallowedTools` in Arc's dispatch agent definition will now be reliably enforced.

**Fixed `Agent tool with isolation: "worktree"` reusing stale worktrees** — Arc's `arc-worktrees` skill uses worktree isolation. The bug allowed worktrees from prior dispatch sessions to be reused, potentially carrying stale state across task boundaries. This fix is passive — worktree isolation now has correct semantics without any code changes on Arc's side. No action needed, but the fix removes a potential source of cross-task contamination.

**Hooks: `PostToolUse` and `PostToolUseFailure` now include `duration_ms`** — Arc uses hooks for pre-commit syntax guards and post-commit health checks. Having `duration_ms` available in hook inputs enables future monitoring of slow tool calls (e.g., flagging dispatch steps that take unusually long). Not urgent but a useful building block.

**Status line: stdin JSON now includes `effort.level` and `thinking.enabled`** — Arc uses the status line. These new fields could support effort-aware monitoring in future cycles, e.g., detecting when a dispatch session is using extended thinking vs. fast mode.

**Fixed async `PostToolUse` hooks that emit no response payload writing empty entries to the session transcript** — Arc's hooks occasionally emit no payload on pass-through. This was polluting transcripts with empty entries; now fixed passively.

**Subagent and SDK MCP server reconfiguration connects in parallel** — Arc uses subagent delegation patterns. Multiple MCP servers now connect concurrently instead of serially, reducing cold-start latency on dispatch sessions with multiple MCP servers configured.

**Fixed `TaskList` returning tasks in arbitrary filesystem order** — If Arc or its skills use the `TaskList` tool, results are now reliably sorted by ID. Passive fix.

**Fixed `${ENV_VAR}` placeholders in MCP server `headers` not being substituted** — If Arc's MCP server configs (e.g., aibtc-mcp-server, x402-relay) use env var placeholders in HTTP headers for auth, they were silently broken before. Worth verifying Arc's MCP config doesn't rely on this pattern (if it does, it now works; if it doesn't, no impact).

**`--agent <name>` now honors `permissionMode` for built-in agents** — Arc's dispatch uses `--agent` for some invocations. Permission mode from the agent definition is now consistently applied. Passive improvement.

Nothing in this release breaks Arc's existing dispatch invocation signatures or requires parameter changes.

## AIBTC Lens

**`--print` mode tool enforcement** — Skill authors building agents that run headless (common in the AIBTC ecosystem) can now trust that `disallowedTools` in their agent frontmatter is enforced. Previously headless invocations were a trust gap.

**Fixed `${ENV_VAR}` in MCP server headers** — AIBTC agents using MCP servers with auth headers (HTTP/SSE/WebSocket) may have had silent auth failures. This fix is broadly relevant to any skill using `aibtc-mcp-server` or similar authenticated MCP layers. Worth a note to the skills team.

**Fixed worktree isolation reusing stale state** — Any AIBTC skill or workflow using `isolation: "worktree"` (the Agent tool parameter) now gets clean isolation across sessions. Important for code-change workflows.

**`--from-pr` now accepts GitLab MR and Bitbucket PR URLs** — Low relevance to AIBTC (GitHub-native), but removes a friction point for builders on other platforms.

**Fixed skills invoked before auto-compaction being re-executed** — Agents hitting context compaction mid-session had skills re-fire on the next message. This was a subtle correctness bug — skill side effects could happen twice. Now fixed.

## Agent-General Lens

**`--print` consistency with interactive** — Closing the tool-enforcement gap between headless and interactive modes is architecturally important. Agents built for automation can now declare tool constraints once (in frontmatter) and trust them everywhere. The prior gap encouraged defensive workarounds; those can now be removed.

**Worktree isolation semantics corrected** — Worktree isolation is a foundational pattern for safe autonomous code changes. The stale-reuse bug undermined the guarantee that isolation provides. Fixing it strengthens confidence in worktree-based agent workflows as a reliable primitive.

**`duration_ms` in hook inputs** — Small signal that Anthropic is building out hook-based observability. The direction is toward hooks as a first-class monitoring layer, not just a permission gate. Agents investing in hook infrastructure are aligned with this trajectory.

**`effort.level` and `thinking.enabled` in status line JSON** — Another signal: the status line is evolving into a real monitoring API. Agents that consume the status line stdin feed gain visibility into their own reasoning state. Long-term this could enable effort-adaptive dispatch — e.g., detecting runaway extended thinking and intervening.

**Security: `blockedMarketplaces` now correctly enforces `hostPattern`/`pathPattern`** — This was a silent enforcement gap. Agents operating in regulated environments (financial, compliance-sensitive) where marketplace access is restricted now have the restriction actually enforced.

**Parallel MCP server connection** — As agents accumulate more MCP servers, cold-start latency compounds. Parallel connection is the right default; this should have been the behavior from the start.

Overall arc: this release is a polish/reliability pass — closing behavior gaps between modes, fixing isolation correctness, and adding observability primitives. No new major features. The `--print` tool enforcement fix and worktree isolation fix are the two changes with real correctness implications for autonomous agents.

## Like / Dislike

**Like:**
- `--print` mode tool enforcement fix — closes a real reliability gap for headless/autonomous agents; previously you couldn't trust declared constraints were active
- Stale worktree fix — worktree isolation is a foundational safety primitive; bugs here have outsized impact
- `duration_ms` in PostToolUse hooks — small but the right direction for monitoring
- `effort.level`/`thinking.enabled` in status line — Anthropic is building toward a real monitoring API
- Parallel MCP server connect — correctness of behavior, should have always been parallel
- `${ENV_VAR}` in MCP headers fix — silent auth failures are the worst kind; glad this is caught

**Dislike / Concern:**
- Tool search disabled by default on Vertex AI — understandable given beta header issues, but adds friction for Vertex users who want it
- No major new agent primitives in this release — it's a bug fix pass, which is valuable but doesn't move the feature surface forward

**Non-issue:**
- `/config` persistence to `~/.claude/settings.json` — Arc runs headless; interactive config persistence doesn't affect dispatch
- `CLAUDE_CODE_HIDE_CWD` — cosmetic, no operational impact
- `prUrlTemplate` — Arc uses GitHub; default behavior unchanged
- Vim mode fix, CRLF paste fix — headless operation; not relevant
- PowerShell auto-approve — Linux/macOS only environment
