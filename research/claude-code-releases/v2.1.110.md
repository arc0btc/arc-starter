# Claude Code v2.1.110 — Applicability Assessment

*Published: 2026-04-15T22:07:18Z*
*URL: https://github.com/anthropics/claude-code/releases/tag/v2.1.110*
*Assessed: 2026-04-15*

## Arc Lens

### Bash Tool Timeout Enforcement [ACTION REQUIRED]

> "Bash tool now enforces the documented maximum timeout instead of accepting arbitrarily large values"

Previously, passing any integer to the Bash tool's `timeout` parameter worked regardless of magnitude. Now it's clamped to the documented maximum (120,000ms / 2 minutes per the tool schema). Any Arc skill that passed large timeout values as a workaround for long-running commands will now hit enforced limits.

Arc runs Claude Code as a subprocess — any Bash tool calls inside dispatched sessions are affected. Skills with long-running shell commands (blockchain RPC calls, file processing, sensor operations) should be audited. **Create a follow-up task to audit skills for over-limit timeout usage.**

### PermissionRequest Hook Fix [VERIFY, LOW RISK]

> "Fixed `PermissionRequest` hooks returning `updatedInput` not being re-checked against `permissions.deny` rules; `setMode:'bypassPermissions'` updates now respect `disableBypassPermissionsMode`"

Given the v2.1.108 saga (where `bypassPermissions` was silently overridden), this fix warrants a quick verification pass. Arc's dispatch uses `--permission-mode bypassPermissions` + `--allow-dangerously-skip-permissions` + `CLAUDE_CODE_SUBPROCESS_ENV_SCRUB=0` — this is distinct from a PermissionRequest hook returning `setMode:'bypassPermissions'`. Should not break Arc's dispatch, but worth confirming after upgrading.

### Non-Streaming Fallback Hang Fix [FREE WIN]

> "Fixed non-streaming fallback retries causing multi-minute hangs when the API is unreachable"

Dispatch was previously vulnerable to multi-minute stalls during API transients. This is now fixed. No action required — free reliability improvement.

### MCP SSE/HTTP Hang Fix [FREE WIN for aibtc-mcp-server]

> "Fixed MCP tool calls hanging indefinitely when the server connection drops mid-response on SSE/HTTP transports"

The `aibtc-mcp-server` uses HTTP transport (v1.47.1). Dropped connections during review operations previously caused silent hangs. This fix directly improves reliability of MCP-dependent dispatch tasks (PR reviews, beat editor operations). No action required.

### Session Cleanup [FREE WIN]

> "Fixed session cleanup not removing the full session directory including subagent transcripts"

Dispatch runs subagent sessions for delegated work. Incomplete cleanup was causing disk bloat over time. Now fixed — no action required.

### SDK/Headless TRACEPARENT Support [FUTURE INTEREST]

> "SDK/headless sessions now read `TRACEPARENT`/`TRACESTATE` from the environment for distributed trace linking"

Arc could inject trace IDs from dispatch cycles into subprocess invocations to link orchestrator + subagent spans. Not urgent but worth noting as a future observability lever.

### Auto-Title Extra Haiku Request Fix [COST SAVINGS]

> "Fixed headless/SDK session auto-title firing an extra Haiku request when `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` or `CLAUDE_CODE_DISABLE_TERMINAL_TITLE` is set"

If Arc sets `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1` (check `.env`), this was causing silent extra Haiku charges per dispatch cycle. Now fixed. **Check whether Arc has this env var set and whether this explains any unexplained cost ticks.**

### `disable-model-invocation` Skill Fix [RELEVANT]

> "Fixed skills with `disable-model-invocation: true` failing when invoked via `/<skill>` mid-message"

Some Arc skills may use this flag. Regression now fixed — no action needed.

## AIBTC Lens

### MCP stdio Stray-Line Fix [RELEVANT for skill authors]

> "Fixed stdio MCP servers that print stray non-JSON lines to stdout being disconnected on the first stray line (regression in 2.1.105)"

A regression from v2.1.105 that disconnected stdio MCP servers on first debug/log line is now fixed. Skill authors implementing MCP servers with any stdout logging can rely on stable behavior again.

### Bash Timeout Enforcement [AFFECTS SKILLS]

Skills in the arc-starter ecosystem that rely on long-running shell commands need to stay within 2-minute Bash timeout. JingSwap P2P fallback, Hiro RPC calls, and blockchain query operations should be checked.

### `--resume`/`--continue` Task Resurrection [INTERESTING]

> "`--resume`/`--continue` now resurrects unexpired scheduled tasks"

For AIBTC agents using scheduled tasks, this means resuming a session can pick up pending scheduled work. Interesting primitive for session-aware task handoff patterns.

## Agent-General Lens

**Tightening runtime contracts.** The Bash timeout enforcement signals Anthropic is hardening previously-lenient behaviors. Agents that relied on undocumented permissive behavior (arbitrarily large timeouts, loose permission propagation) will need to stay within documented contracts. Expect more of this.

**Distributed tracing as a first-class primitive.** TRACEPARENT/TRACESTATE support in headless sessions means Anthropic is thinking about agent orchestration observability. Multi-agent systems can now propagate trace context through subprocess invocations — good infrastructure signal.

**MCP reliability hardening.** Two MCP fixes in this release (SSE/HTTP hang, stdio stray-line regression). Anthropic is clearly investing in MCP as a stable integration layer, not just a demo feature. Agents building MCP-backed skills are on a supported path.

**TUI improvements are noise for headless agents.** The `/tui`, `autoScrollEnabled`, `Ctrl+O` changes are all interactive UI features. Zero relevance for dispatch-mode agents. Don't chase UI config settings in agent contexts.

## Like / Dislike

**Like:**
- Non-streaming hang fix — was a silent dispatch reliability killer, now fixed for free
- MCP SSE/HTTP hang fix — directly benefits Arc's aibtc-mcp-server integration
- Session cleanup fix — disk hygiene for long-running agent deployments
- TRACEPARENT support — forward-looking observability primitive

**Dislike / Concern:**
- Bash timeout enforcement without explicit migration guidance — agents that relied on large timeouts will fail silently (the Bash call just won't run as long as expected). No deprecation warning path.
- `bypassPermissions` + `disableBypassPermissionsMode` interaction change — requires manual verification that Arc's dispatch workaround from v2.1.108 still holds. The v2.1.108 saga cost real money; any regression here would be expensive.

**Non-issue:**
- All TUI/UI changes (`/tui`, `autoScrollEnabled`, `Ctrl+O`, `autoScrollEnabled`, focus mode) — irrelevant for headless dispatch
- Remote Control fixes — Arc doesn't use Remote Control
- Plugin install dependency fix — Arc doesn't use the plugin marketplace
- Session recap for DISABLE_TELEMETRY users — not applicable to Arc's deployment
