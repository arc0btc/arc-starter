# Claude Code v2.1.81 — Applicability Assessment

## Arc Lens

### `--bare` flag — do NOT use for Arc dispatch

The new `--bare` flag skips hooks, LSP, plugin sync, and skill directory walks for scripted `-p` calls. At first glance this looks attractive for Arc's dispatch subprocess — less overhead, cleaner execution. But Arc's dispatch resilience is built on two hook-based safety layers:

1. **Pre-commit syntax guard** — Bun transpiler validates staged `.ts` files before committing. A syntax error blocks the commit and creates a follow-up task.
2. **Post-commit service health check** — After committing `src/` changes, snapshots service state; reverts and creates follow-up if a service died.

Using `--bare` would silently bypass both. Arc must NOT use `--bare` for main dispatch cycles. This decision should be documented to prevent future drift — the flag is tempting because Arc already requires `ANTHROPIC_API_KEY` (satisfies the flag's auth requirement) and disabling auto-memory aligns with Arc's explicit MEMORY.md approach.

Possible narrow use: if Arc ever spawns ultra-lightweight one-shot analysis tasks that never touch `src/`, `--bare` could reduce cycle startup overhead. Not worth the risk today.

### Background agent task hang fix — directly relevant to Arc

> Fixed race condition where background agent task output could hang indefinitely

Arc's memory flags an active risk: "Running v2.1.73, below required v2.1.79 (stdin hang fix). Task #7215 failed to verify upgrade." This v2.1.81 fix closes an additional class of agent hang. **Upgrading to v2.1.81 resolves both the stdin hang (v2.1.79) and this background task race condition.** Upgrade is now more urgent than the original task framing.

### Worktree session resume fix

> Resuming a session that was in a worktree now switches back to that worktree

Arc uses worktree isolation via the `arc-worktrees` skill. If dispatch is ever interrupted and resumed mid-worktree task, the session will now correctly re-enter the isolated worktree rather than running from main. This is a free reliability improvement for Arc's worktree workflow.

### Permission prompts for Bash commands with dashes

> Fixed unnecessary permission prompts for Bash commands containing dashes in strings

Arc skills frequently run CLI commands with flags (e.g., `arc tasks add --subject "..." --priority 5`). If any of these were triggering spurious permission prompts, this fix silently cleans that up. No action needed.

### `CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS` fix

Arc runs against the standard Anthropic API, not a proxy. This fix (400 errors on proxy gateways when structured-outputs beta header wasn't suppressed) doesn't affect Arc today. Relevant if Arc ever routes through a proxy or Bedrock/Vertex.

### `--channels` permission relay

v2.1.80 introduced `--channels` as MCP push messaging (research preview). v2.1.81 extends it: channel servers that declare the permission capability can forward tool approval prompts to a phone. For Arc, this is architecturally interesting — it could replace the "set status=blocked and wait" pattern for irreversible actions. Instead of blocking the queue, Arc could push an approval request and pause just that task. Not production-ready; worth tracking for the escalation workflow redesign.

## AIBTC Lens

### `--bare` for skill authors building scripted pipelines

Skill authors building automation that calls `claude -p` can now use `--bare` for clean, lightweight execution — no hooks overhead, no skill directory walks, no auto-memory, no plugin sync. This is a meaningful ergonomic improvement for building tools that invoke Claude Code programmatically. arc-starter `AGENT.md` documentation should mention `--bare` as the recommended flag for scripted subagent invocations that don't need hook safety layers.

### MCP OAuth with CIMD/SEP-991

> Improved MCP OAuth support — Updated to support Client ID Metadata Document (CIMD / SEP-991) for servers without Dynamic Client Registration

The `aibtc-mcp-server` v1.41.0 integration review (task #7594) flagged MCP as critical path for Arc. SEP-991/CIMD broadens the range of MCP servers Claude Code can authenticate against — servers that don't implement Dynamic Client Registration are now supported. This matters for AIBTC's MCP ecosystem as it grows beyond the reference implementation.

### Improved MCP tool call display

> MCP read/search tool calls collapse into a single "Queried {server}" line (expand with Ctrl+O)

Cleaner output when agents use MCP tools extensively. Better developer experience when debugging arc-starter sessions that invoke multiple MCP tools.

### Plugin freshness (ref-tracked plugins re-clone on every load)

If AIBTC skills are ever distributed as ref-tracked Claude Code plugins, updates are now picked up on every load without manual refresh. Lower maintenance burden for the skills ecosystem.

## Agent-General Lens

**`--bare` is Anthropic's formal endorsement of headless agent mode.** The flag's design is deliberate: require API key (not OAuth), disable auto-memory (agents manage their own memory), disable hooks and plugin infrastructure (agent orchestrators handle these). This is Anthropic saying: "we know you're running Claude Code in automated pipelines, here's a mode optimized for that." Agents building on top of Claude Code should evaluate whether `--bare` fits their pipeline — but understand what they're giving up (hooks = safety nets, skill walks = context loading).

**Permission relay is the supervision architecture taking shape.** v2.1.80 introduced MCP push channels. v2.1.81 activates it for permission relay — forwarding approval prompts to a human's phone. The pattern: autonomous agent runs, hits a decision requiring human judgment, pauses and pushes a notification, resumes after approval. This is a proper human-in-the-loop primitive, not just "set blocked and wait." If this lands as stable infrastructure, it changes how agents handle irreversible-action escalation.

**Multi-agent reliability investment continues.** Background task hang race condition fixed. Worktree session resume fixed. Concurrent session re-auth fixed. The consistent pattern across recent releases: Anthropic is actively fixing the failure modes you hit when running Claude Code as an orchestrated multi-agent system, not just a single interactive session. This investment signals Claude Code is intended as production agent infrastructure.

**MCP as the integration layer is deepening.** CIMD/SEP-991 support, better display, push channels, permission relay through channels — all of this is MCP-adjacent. The direction is clear: MCP is not a "tool calling" convenience, it's becoming the runtime fabric for agent-to-server and agent-to-human communication.

## Like / Dislike

**Like:**
- `--bare` is the right abstraction for scripted agent use cases — clean, explicit, no hidden state
- Background task hang fix is directly relevant to Arc's dispatch reliability
- Worktree session resume fix is a quiet but valuable improvement for Arc's isolation model
- Permission relay direction is exactly right for escalation workflows
- MCP CIMD/SEP-991 broadens integration surface meaningfully

**Dislike / Concern:**
- `--bare` is easy to misuse — skipping hooks in dispatch would silently remove safety nets. The flag needs explicit documentation in Arc's dispatch config to prevent future temptation.
- `--channels` permission relay is still not production-stable (no GA timeline visible). Hard to plan the escalation workflow redesign around it.

**Non-issue:**
- Voice mode fixes — no voice use in headless dispatch
- Node.js 18 crash fix — Arc runs on Bun
- Remote Control session titles, `/rename`, `/exit` — interactive session features, not relevant for automated dispatch
- Windows PATH inheritance fix — Linux deployment
