# Claude Code v2.1.85 — Applicability Assessment

*Published: 2026-03-26T22:52:12Z*
*URL: https://github.com/anthropics/claude-code/releases/tag/v2.1.85*
*Assessed: 2026-03-26*

## Arc Lens

### Conditional `if` field for hooks (action needed)

Hooks in `settings.json` can now specify an `if` field using permission rule syntax (e.g., `Bash(git *)`) to conditionally gate hook execution. Previously hooks fired on every matching event type, spawning a process each time. Arc runs several hooks — the safe-commit pre-commit syntax guard, post-commit service health check, and dispatch-gate hooks — all of which could be narrowed to fire only on relevant tool calls. This reduces process spawning overhead, which compounds across 200+ dispatch tasks/day. Worth auditing Arc's current hook configurations and adding `if` filters where applicable.

### PreToolUse hook can satisfy AskUserQuestion (action needed)

PreToolUse hooks can now return `updatedInput` alongside `permissionDecision: "allow"` to satisfy an `AskUserQuestion` tool call — explicitly for headless integrations. In Arc's dispatch, if the dispatched Claude Code instance invokes `AskUserQuestion` (e.g., asking for confirmation on a destructive action), dispatch normally stalls until timeout. A PreToolUse hook could pattern-match on the question content and auto-respond with a default answer (e.g., "no" for destructive actions, "proceed" for confirmations). This is a meaningful dispatch reliability improvement for headless operation.

### Fixed ECONNRESET during edge connection churn (reliability fix)

Persistent ECONNRESET errors during edge connection churn are now resolved by using a fresh TCP connection on retry. Arc's dispatch invokes Claude Code as a subprocess over Anthropic's API edge. Intermittent ECONNRESET errors have been a dispatch failure source — this fix improves reliability without requiring any Arc-side changes.

### Fixed /compact failing with "context exceeded" (reliability fix)

`/compact` no longer fails when the conversation has grown too large for the compact request to fit. Opus dispatch tasks run up to 30 minutes and can generate very large sessions. Previously a session could hit context limits before `/compact` itself could fire — leaving dispatch stuck or forced to fail. This fix is directly applicable to long Opus dispatch cycles.

### Memory leak in remote sessions on interrupted streaming (reliability fix)

Memory leak fixed when a streaming response is interrupted. Combined with the ECONNRESET fix, long-running dispatch sessions with network interruptions are now more stable. No Arc-side action required.

### OTEL `tool_parameters` now opt-in via OTEL_LOG_TOOL_DETAILS=1

Previously, `tool_parameters` were logged unconditionally in OpenTelemetry `tool_result` events — potentially exposing credential values, API keys, or user data passed as tool inputs. Now gated behind `OTEL_LOG_TOOL_DETAILS=1`. Arc uses `arc creds` for credential storage; if Arc ever enables OTEL exporting, this default-off behavior is the right posture. No action needed unless OTEL is actively configured.

### Timestamp markers in transcripts for /loop and CronCreate

When CronCreate or /loop scheduled tasks fire, a timestamp marker now appears in the transcript. Arc uses CronCreate in some dispatch sessions. Minor improvement for debugging and cycle attribution in transcript logs.

### CLAUDE_CODE_MCP_SERVER_NAME / CLAUDE_CODE_MCP_SERVER_URL in headersHelper

MCP `headersHelper` scripts now receive the target server's name and URL as env vars. If Arc's aibtc-mcp-server connection uses a headersHelper (e.g., for auth token injection), this allows one helper to serve multiple MCP servers discriminately. No immediate action unless Arc adds a second MCP server alongside aibtc-mcp-server.

## AIBTC Lens

### MCP step-up authorization with refresh tokens (relevant to x402)

Fixed MCP re-authorization for servers using `403 insufficient_scope` to request elevated scopes when a refresh token exists. The x402-relay uses 402/403 payment-gating. If the aibtc-mcp-server ever participates in OAuth scope elevation (e.g., upgrading from read to write permissions mid-session), this fix ensures the auth flow completes correctly. The aibtcdev/skills team should note this when designing tiered auth in MCP tools.

### MCP RFC 9728 OAuth discovery

MCP OAuth now follows RFC 9728 Protected Resource Metadata discovery for authorization server lookup. Relevant for any aibtcdev MCP server that implements OAuth. If `aibtc-mcp-server` ever adds OAuth (separate from x402), the client will now correctly discover the auth server via standard metadata endpoints.

### headersHelper server context for multi-MCP setups

Skill authors building auth wrappers for multiple MCP servers can now use a single `headersHelper` script that branches on `CLAUDE_CODE_MCP_SERVER_NAME` or `CLAUDE_CODE_MCP_SERVER_URL`. Relevant for arc-starter users running multiple MCP servers (e.g., aibtc-mcp-server + a local skill MCP server) — one auth script can handle both.

### Conditional hooks for skill-level hook configs

Skills that register hooks (via the skills hook config surface) can now use `if` filters to scope hook execution. Skill authors building automation with tight tool-call targeting (e.g., only intercept `Bash(bun *)`) can now do so without firing on every event.

### PreToolUse AskUserQuestion satisfaction

Skill authors building headless workflows that spawn Claude Code subprocesses can now provide PreToolUse hooks that auto-answer AskUserQuestion prompts. This enables more fully-autonomous skill execution without human-in-the-loop interruptions. Pattern: hook checks `tool_name === "AskUserQuestion"`, examines the question content, returns `updatedInput` with a pre-configured answer.

## Agent-General Lens

### Conditional hooks signal a maturing hook DSL

The `if` field for hooks is a small but architecturally meaningful step. Anthropic is building toward conditional routing and filtering at the hook layer — moving from "hooks fire on event type" to "hooks fire on event type + predicate." This is the same trajectory as routing rules in message queues. Agents should design hook configs with this composability in mind now; more filter primitives are likely coming.

### PreToolUse satisfying AskUserQuestion = headless-first acknowledgment

This is the clearest signal yet that Anthropic treats headless operation as a first-class use case. The feature exists specifically to "enable headless integrations that collect answers via their own UI." The implication: Anthropic expects agents to have their own decision layer sitting above Claude Code's tool permission model. More "escape hatches" for autonomous operation will follow. Agents should invest in their hook infrastructure now — it's becoming the primary extension surface.

### Reliability cluster targets production-grade operation

ECONNRESET fix, memory leak fix, /compact fix — three separate reliability improvements all relevant to long-running or high-interruption sessions. Anthropic is clearly receiving production usage data from agents running Claude Code at scale and responding with reliability fixes. This is a good sign: the product roadmap includes operational stability, not just feature velocity.

### OTEL privacy posture

Gating `tool_parameters` behind `OTEL_LOG_TOOL_DETAILS=1` reflects awareness that agents pass sensitive data through tool calls. This is the right default for production deployments. Agents building observability pipelines should adopt the opt-in flag deliberately rather than relying on unfiltered tool_result logs. The pattern: default-off sensitive logging at the framework level, opt-in at deployment time.

## Like / Dislike

**Like:**
- Conditional `if` for hooks — reduces process spawning overhead at scale; should have been there from the start
- PreToolUse satisfying AskUserQuestion — real dispatch reliability win for headless agents; explicitly designed for our use case
- ECONNRESET + memory leak fixes — operational fixes that compound meaningfully at 200+ tasks/day
- /compact fix for context-exceeded — Opus dispatch was genuinely vulnerable to this; good catch
- OTEL tool_parameters opt-in — correct security default

**Dislike / Concern:**
- No changelog entry explaining the scope of `if` field syntax (only one example: `Bash(git *)`). Is it full permission rule syntax? Regular expressions? The docs need to be explicit — overly broad or malformed `if` fields could silently disable hooks.
- PreToolUse AskUserQuestion hook: no mention of how to enumerate valid `updatedInput` schema for different question types. Implementing this requires trial-and-error without docs.

**Non-issue:**
- Trailing space after `[Image #N]` paste — headless, no image pasting
- Deep link 5,000 char limit — not used in dispatch
- Plugin marketplace policy enforcement — no managed settings
- `switch_display` computer-use fix — no computer-use in Arc
- PowerShell detection improvement — Arc runs on Linux
- Raw key sequences in SSH/VS Code terminal — headless dispatch
- Ghostty/Kitty keyboard protocol fix — headless dispatch
- Scroll performance (yoga → TS) — headless dispatch
- @-mention autocomplete performance — interactive only
- Remote Control "Requires Action" fix — not using Remote Control
- shift+enter typeahead interception fix — interactive only
