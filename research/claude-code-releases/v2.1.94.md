# Claude Code v2.1.94 — Applicability Assessment

*Published: 2026-04-07T21:18:51Z*
*URL: https://github.com/anthropics/claude-code/releases/tag/v2.1.94*
*Assessed: 2026-04-07*

## Arc Lens

### Changed: Default effort level medium → high for API-key users

**Directly relevant — cost impact.** Arc dispatch invokes Claude Code as a subprocess using an API key. Starting with v2.1.94, the default effort level changes from medium to high for API-key users. Arc's `dispatch.ts` does not currently set an explicit effort level. For non-opus models, `MAX_THINKING_TOKENS=10000` is set, which limits the thinking budget regardless of effort level. But for **opus** tasks, no thinking token cap is set — meaning opus dispatch cycles now operate at the higher default effort, consuming more tokens and incurring higher cost per cycle.

The `/effort` control is interactive-only (slash command), not a CLI flag. Investigate whether `MAX_THINKING_TOKENS` fully constrains effort-level-driven token usage for opus, or whether Arc needs an additional pin. This is a cost-relevant config gap.

No config change deployed yet — follow-up task created (see below).

### Fixed: Agents stuck after 429 with long Retry-After

**Directly relevant.** When Claude Code hit a rate limit with a long `Retry-After` header (e.g., 60+ seconds), it silently waited rather than surfacing the error. This could make a dispatch cycle appear stuck — burning the dispatch timeout window without producing output, and leaving the dispatch lock held. The fix surfaces the error immediately instead of waiting. Arc's dispatch already has timeout detection and retry logic; this fix means rate-limited cycles now fail fast and return a retryable error rather than hanging until the outer watchdog fires.

### Fixed: SDK/print mode partial response not preserved when interrupted mid-stream

**Moderately relevant.** Arc dispatch runs Claude Code in `--print` (SDK/print) mode and captures the stream-JSON output. If a dispatch cycle is interrupted mid-stream — by SIGKILL from the watchdog timeout or by a forced stop — the partial assistant response was not preserved in conversation history. The fix ensures the partial response is retained even on interrupted cycles. For dispatch cycles that time out mid-task, this means better context recovery on subsequent resumptions.

### Improved: `--resume` from other worktrees works directly

**Useful for arc-worktrees.** Previously, resuming a session from a different worktree of the same repo printed a `cd` command instructing the user to change directories before resuming. Now it resumes directly. Arc's worktree isolation skill (`arc-worktrees`) creates temporary git worktrees per task. If a worktree task needs to be resumed, this improvement removes a manual step.

### Fixed: CJK and multibyte UTF-8 corruption in stream-json

**Directly relevant.** Arc dispatch uses `--output-format stream-json`. If any content in the output contained multibyte UTF-8 (agent names, BNS names, contract addresses with Unicode notes, commit messages in non-ASCII) and chunk boundaries happened to split a UTF-8 sequence, the output was corrupted with U+FFFD replacement characters. This could silently corrupt `result_summary` or `result_detail` stored in the task DB. Fixed without config change.

### Added: `hookSpecificOutput.sessionTitle` to UserPromptSubmit hooks

**Minor — useful quality-of-life.** Arc's UserPromptSubmit hook (`session-start.sh`) could emit `sessionTitle` to label each dispatch session with the task ID and subject. Currently, dispatch cycles show as generic sessions in logs. This is a low-priority improvement — no operational need, but useful for debugging long-running dispatch logs.

### Non-issues for Arc

- **Bedrock Mantle** (`CLAUDE_CODE_USE_MANTLE=1`) — Arc uses Anthropic direct API. Non-issue.
- **`keep-coding-instructions` frontmatter** — Arc's dispatch doesn't use Claude Code plugins. Non-issue.
- **Plugin skills using frontmatter `name`** — Arc's skill system is separate from Claude Code's plugin marketplace system. Arc's `skills/<name>/` directories are not Claude Code plugins. Non-issue.
- **Plugin hook fixes** (YAML frontmatter silently ignored, `CLAUDE_PLUGIN_ROOT` errors) — Arc doesn't use Claude Code plugin hooks. Non-issue for dispatch.
- **macOS/VSCode/UI fixes** — Arc runs headless on Linux. Non-issues.
- **Slack MCP compact header** — Arc doesn't use the Slack MCP tool. Non-issue.

## AIBTC Lens

### Plugin skill YAML frontmatter hooks were silently ignored — now fixed

**High impact for marketplace skill authors.** The aibtcdev/skills marketplace publishes skills as Claude Code plugins. Plugin skills can define hooks in YAML frontmatter. Prior to v2.1.94, these hooks were **silently ignored** — meaning any marketplace skill with hook behavior (pre/post tool-use guards, custom output handlers) was not executing those hooks at all. This is a significant behavioral fix. Skills that relied on frontmatter hooks for access control, logging, or output shaping were operating without those controls. Marketplace skill authors should audit their hooks to verify behavior post-upgrade.

### Plugin invocation name now stable (uses frontmatter `name`, not directory basename)

**Relevant for skill consumers.** AIBTC builders installing marketplace skills via `"skills": ["./"]` now get a stable invocation name from the skill's frontmatter `name` field, regardless of how the directory is named or installed. Previously, the invocation name derived from the directory basename, which could vary across install methods. This means skill documentation and agent prompts can reference skills by their canonical frontmatter name — more stable for multi-agent workflows.

### Default effort high — cost impact for builders on API keys

AIBTC developers running Claude Code interactively or programmatically with an API key will see higher default token usage. Builders using arc-starter patterns for their own agents should audit their dispatch invocations and decide whether to pin effort level explicitly. The quality-vs-cost tradeoff is now shifted toward quality by default.

### Bedrock Mantle support

AIBTC builders deploying on AWS infrastructure have a new Bedrock variant option. Broader deployment flexibility for the ecosystem.

## Agent-General Lens

### Effort default shift signals quality-first philosophy for authenticated users

Anthropic is drawing a clear tier distinction: free/lite users get medium effort; API-key and enterprise users get high effort by default. The implicit message is that paying users are expected to want more compute, not less. For autonomous agents with usage cost sensitivity, this default will need explicit override. The pattern to watch: effort level may become a per-task config variable, not just a global default. Agents that run heterogeneous tasks (some requiring deep reasoning, others trivial) should explicitly set effort per task rather than relying on the global default.

### 429 fast-fail is correct autonomous agent design

Silent retry-waiting on long Retry-After headers is a category of failure that disproportionately affects autonomous agents over interactive users. A human notices their session froze; an autonomous dispatch loop doesn't — it just wastes its timeout window. The fix (surface immediately, let the caller decide retry policy) is the right abstraction boundary. The lesson: agent runtimes should never silently absorb time-consuming wait states. Fast failure + caller-side retry is always preferable to runtime-level silence.

### SDK/print mode interruption recovery points toward resumable dispatch

Preserving the partial response on mid-stream interruption is a small step toward resumable agent sessions. For long-running autonomous tasks, the ability to resume from a known-good partial state (rather than restart) is significant. Anthropic appears to be investing in making the `--print` mode (the standard agent invocation path) more resilient to interruption. Watch for explicit `--resume` support in `--print` mode in future releases.

### Hook output is evolving toward structured metadata, not just pass/fail

The `hookSpecificOutput.sessionTitle` addition is small, but it continues a pattern: hooks are gaining the ability to inject structured metadata into Claude Code's session context, not just block or allow tool calls. This signals a richer hook model ahead — hooks may eventually be able to annotate sessions, set context variables, or inject structured facts into the conversation. Agents using hooks for more than simple pass/fail control should track this evolution.

## Like / Dislike

**Like:**
- 429 fast-fail — prevents the category of silent dispatch hangs from rate limit waits. Exactly right.
- UTF-8 stream-json fix — Arc uses stream-json; corrupted multibyte content in result fields was an invisible data quality issue.
- Partial response preservation on interruption — dispatch cycles that hit the timeout wall now have a better chance of recovering context.
- Plugin YAML frontmatter hooks fix — this was a silent contract violation for marketplace skill authors; now it works as documented.

**Dislike / Concern:**
- Default effort high without CLI flag to control it programmatically — `/effort` is interactive only. Arc dispatch can't set effort level via a flag in `--print` mode. The only current lever is `MAX_THINKING_TOKENS` env var, which is set for non-opus but not opus. The effort level change could silently inflate opus cycle costs without a clean override path. Needs investigation.

**Non-issue:**
- Bedrock Mantle — Arc uses Anthropic direct.
- Plugin invocation name stability — Arc's skill system is not Claude Code plugins.
- Plugin PLUGIN_ROOT fixes — Arc doesn't use `CLAUDE_PLUGIN_ROOT`.
- VSCode fixes — Arc is headless Linux.
- macOS keychain login fix — Linux deployment.
- Slack MCP compact header — Arc doesn't use Slack MCP.
- Shift+Space, hyperlink double-tab, alt-screen ghost lines — UI bugs, headless non-issues.
- `keep-coding-instructions` frontmatter — not applicable to dispatch.
