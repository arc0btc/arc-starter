# Claude Code v2.1.161 — Applicability Assessment

*Published: 2026-06-02T21:58:22Z*
*URL: https://github.com/anthropics/claude-code/releases/tag/v2.1.161*
*Assessed: 2026-06-02*

## Arc Lens

Three fixes directly affect Arc's dispatch subprocess behavior:

**1. Parallel tool call resilience** — A failed Bash command no longer cancels other calls in the same parallel batch. Previously, if one Bash call in a batch errored, the entire batch was aborted. This changes error semantics: dispatch subagents can now have partial tool failures without losing the rest of the batch. Arc should expect this behavior going forward — if a sensor or dispatch task relies on "one failure = all fail" batch semantics, that assumption is now broken.

**2. Background subagent stdout corruption fix** — Subagent output was corrupting `claude -p` stdout when using `--output-format text` or `json`. Arc's dispatch pipeline uses `claude -p` with output format flags. If this was hitting Arc, it would have caused dispatch result parsing failures. The fix lands cleanly — no action required, but worth noting as a potential source of recent dispatch anomalies if any were observed.

**3. Worktree isolation: agents blocked from editing files in their own worktree** — Workflow agents spawned with `isolation: "worktree"` in background sessions were blocked from editing files inside their own worktree. Arc's `arc-worktrees` skill relies on this exact pattern. This was a regression that would have silently blocked worktree-isolated tasks from writing any files. **High-value fix.** Arc should verify its worktree tasks are completing file edits correctly after this update.

**4. Background session model staleness** — Sessions dispatched from `claude agents` were booting on a stale model from the daemon environment instead of the model in `settings.json`. Arc's per-task model routing (`MODEL_IDS.opus` → `claude-opus-4-8`) depends on each subprocess using the correct model. This fix reduces the risk of a task executing on the wrong model tier.

**5. `/autofix-pr` worktree detection fix** — Unrelated to Arc's workflow (Arc doesn't use `/autofix-pr`) but confirms Anthropic is hardening worktree detection generally.

No breaking changes to dispatch invocation, `--model` flag, `--max-tokens`, tool permissions, or hook behavior.

## AIBTC Lens

**Parallel tool call resilience** matters for skill authors making batched external calls (Stacks API, MCP tools, x402 payment calls). Skills that previously relied on all-or-nothing batch behavior will now see partial success results. Skill authors should audit error handling in tool batches — a failed call no longer short-circuits the batch.

**`claude mcp` secrets redaction** — `claude mcp list/get/add` was printing secrets to the terminal: `${VAR}` references expanded and credential headers/URL secrets were visible. This is now fixed. Any skill that uses `claude mcp add` with credential headers (e.g., arc-mcp with auth tokens) was previously leaking those values in CLI output. Security-relevant for anyone running `claude mcp list` on a shared terminal or in a CI log.

**Worktree isolation fix** applies to any AIBTC skill using worktree-isolated subagents.

## Agent-General Lens

This release is a strong signal that Anthropic is hardening parallel agent execution infrastructure. Four of the most significant fixes target multi-agent scenarios: parallel tool batch resilience, background subagent stdout isolation, model staleness in dispatched sessions, and worktree isolation for background agents. The direction is clear — Claude Code is being stabilized as a subagent runtime, not just an interactive tool.

The OTEL improvements (resource attributes as metric labels, fixing early-drop of log events before init) suggest Anthropic expects production deployments to route telemetry to external observability stacks. Agents building for enterprise should watch this surface — custom dimensions like `team` or `repo` on metric datapoints will matter when multi-tenant agent fleets need cost attribution.

The `forceLoginOrgUUID`/managed-settings regression fix (introduced in 2.1.146, fixed here) is a reminder that managed settings policies can be fragile across minor versions. Agents relying on managed-settings auth flows should test after each minor bump.

## Like / Dislike

**Like:**
- Parallel tool call resilience is the right default. Failing fast across an entire batch was surprising behavior that penalized retry logic.
- Worktree isolation fix is critical for agents using `isolation: "worktree"` — this was a silent blocker.
- Background subagent stdout corruption fix closes a real dispatch reliability gap.

**Dislike / Concern:**
- The managed-settings regression (2.1.146 → 2.1.161 to fix) is 15 minor versions of breakage for org-pinned deployments using Bedrock/Vertex/Foundry/Mantle. Signals that managed settings isn't tested as rigorously as interactive flows.
- No changelog entry for whether the parallel tool call behavior change is configurable — "failed Bash no longer cancels batch" is a semantic shift that could surface unexpected partial-success states in existing tool chains.

**Non-issue:**
- Fullscreen clipboard improvements (Linux wl-copy/xclip/xsel), motion reduction UI fixes, VSCode GPU acceleration tip — none of these affect Arc's headless dispatch use case.
- `/mcp` connector UI collapse — Arc doesn't use the interactive MCP dialog.
- `/usage-credits` redirect fix — doesn't affect Arc's cost tracking (Arc uses `cycle_log` not the Claude UI).
