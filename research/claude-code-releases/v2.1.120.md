# Claude Code v2.1.120 — Applicability Assessment

*Published: 2026-04-25T00:14:31Z*
*URL: https://github.com/anthropics/claude-code/releases/tag/v2.1.120*
*Assessed: 2026-04-25*

## Arc Lens

**`${CLAUDE_EFFORT}` in skill content** — Skills can now reference the current effort level dynamically. For Arc, this means dispatch-loaded SKILL.md files can adapt their instructions based on the effort setting passed to the Claude Code subprocess. Concrete use case: a skill that normally provides verbose step-by-step guidance could say "if effort=low, skip exhaustive source validation." Not using this yet — but it's a primitive worth evaluating for high-context skills like bitcoin-macro and arc-workflows.

**`AI_AGENT` env var for subprocesses** — Claude Code now sets `AI_AGENT=1` for child processes so tools like `gh` can attribute traffic to Claude Code. Arc's dispatch subprocess spawns `gh` for PR creation, issue comments, and release watching. This is a passive attribution improvement — no action required, but good to know `gh` calls are now properly attributed in API/telemetry logs.

**Fixed `find` Bash tool FD exhaustion (host-wide crash)** — Critical stability fix. The Bash tool using `find` on large directory trees could exhaust open file descriptors, causing host-wide crashes on macOS/Linux native builds. Arc's sensors and dispatch sessions do use `find` (Glob tool calls translate to `find` under the hood in some cases). This was a latent crash risk. Update is urgent on the Arc host.

**Fixed false-positive "Dangerous rm operation" in auto mode** — Multi-line bash commands containing both a pipe and a redirect were triggering spurious dangerous-rm permission prompts in auto mode. Arc runs in auto mode; this false positive could have stalled dispatch sessions requiring manual intervention. Fix removes a source of phantom blocking.

**Fixed `DISABLE_TELEMETRY` / `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC` not suppressing usage metrics** — If Arc's environment has telemetry disabled, this was silently failing for API and enterprise users. Check `~/.claude/settings.json` or env vars — if Arc has these set, the fix makes them actually work.

**Fixed MCP stdio bug (Esc closing entire server)** — Regression from 2.1.105 fixed. If Arc uses any MCP servers via stdio transport, pressing Esc during a tool call was terminating the whole server connection. Not a pressing issue for dispatch (non-interactive), but relevant if Arc uses interactive sessions with MCP tools.

**No breaking changes** to `--model`, `--max-tokens`, dispatch subprocess invocation, or any flag Arc currently uses.

## AIBTC Lens

**`${CLAUDE_EFFORT}` in skills** — Skill authors building on arc-starter can now write effort-aware SKILL.md files. Example: a skill could provide abbreviated instructions for haiku-dispatched tasks and full instructions for opus-dispatched tasks. This is a new ergonomic primitive for the skills pattern — worth documenting in AIBTC skill authoring guidance.

**`claude ultrareview [target]` non-interactive subcommand** — Runs `/ultrareview` from CI or scripts without an interactive session, printing findings to stdout (with `--json` for structured output) and exiting 0/1. For AIBTC's PR workflow this means `ultrareview` can be added to GitHub Actions — no longer requires a live session. Arc's current PR workflow runs `/simplify` before PRs; `ultrareview` is a deeper quality pass that could slot in at the same stage or in CI.

**`AI_AGENT` attribution** — Any AIBTC agent using `gh` in Claude Code subprocesses will now have traffic attributed. Good for ecosystem-level visibility of agent activity.

## Agent-General Lens

**Effort-aware skills are the signal** — `${CLAUDE_EFFORT}` is subtle but directionally important. Anthropic is making the agent's internal configuration visible to its own skills. This is a step toward skill-level self-knowledge — skills that behave differently based on how hard the agent is working. The trajectory: agent context eventually includes more runtime state that skills can read and adapt to.

**Non-interactive ultrareview signals CI-first agent tooling** — The move to make `ultrareview` a proper subcommand (not just a slash command) signals Claude Code is being designed as a CI primitive, not just a developer assistant. Agents that run in automated pipelines get first-class review tooling. This is the same direction as `--print` / `--output-format stream-json` — everything becoming scriptable.

**The `find` FD bug was a silent availability risk** — The fact that heavy `find` usage could crash the host OS is a reminder that Claude Code's Bash tool is a system-level primitive, not sandboxed. Long-running agents that do deep filesystem work (sensors scanning large trees, skills running `find` in large repos) have been silently accumulating OS-level risk. Defense: prefer Glob over Bash+find where possible.

**`AI_AGENT` env var as ecosystem attribution primitive** — Third-party tools that check for `AI_AGENT` can now differentiate Claude Code invocations from human-driven ones. This enables agent-aware tooling without requiring out-of-band signals. Early, but it's the pattern by which the agent ecosystem starts to self-describe.

## Like / Dislike

**Like:**
- `${CLAUDE_EFFORT}` — finally a runtime variable skills can use for self-adaptation. Small but architecturally meaningful.
- Non-interactive `ultrareview` subcommand — unblocks CI integration of deep code review.
- `find` FD exhaustion fix — critical stability fix for any long-running agent on Linux/macOS. Urgent update.
- False-positive rm permission fix in auto mode — removes a class of phantom dispatch stalls.
- `DISABLE_TELEMETRY` actually working now — privacy guarantees were silently broken; fixed.

**Dislike / Concern:**
- No structured changelog format (semver breaking vs non-breaking) — impossible to know if any of these are silent regressions without reading every word.

**Non-issue:**
- Windows PowerShell fallback — Arc runs on Linux systemd. Zero relevance.
- Spinner tips changes, scroll hints, VSCode-specific fixes — UI polish, no dispatch impact.
- Faster session start with many claude.ai connectors — Arc doesn't use claude.ai connectors.
