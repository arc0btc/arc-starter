## What Changed

Single commit. One file changed. Empty release body (no changelog text published).

**Commit:** `feat(code-review): pass confirmed=true when posting inline comments`
**PR:** anthropics/claude-code#33472 (Kashyap Murali)
**File changed:** `plugins/code-review/commands/code-review.md`

The code-review plugin prompt now instructs Claude to pass `confirmed: true` when calling `mcp__github_inline_comment__create_inline_comment`. Without this flag, calls are buffered and do not post. This prevents subagents that inherit the tool from accidentally posting test/probe comments to customer PRs. Backward compatible — older claude-code-action versions that don't recognize the param ignore it and post as before.

## AIBTC Lens

**Relevant if using claude-code-action with the bundled code-review plugin.**

Any AIBTC ecosystem builder running claude-code-action for automated PR reviews should verify their setup after this update. The behavioral change:

- Old behavior: calling `mcp__github_inline_comment__create_inline_comment` without `confirmed` posts the comment immediately.
- New behavior: calling without `confirmed: true` buffers the call — the comment is not posted.

If a custom code-review workflow extends or overrides the bundled `code-review.md` prompt without including `confirmed: true`, inline comments will silently stop posting after upgrading to v2.1.75. The fix is trivial (add the param) but the failure mode is silent, which makes it easy to miss.

The broader signal: if any arc-starter-based agent uses the inline-comment MCP tool, check whether it passes `confirmed: true`.

## Agent-General Lens

**Good safety pattern: buffer-by-default for irreversible side effects.**

The `confirmed: true` requirement codifies a principle worth generalizing: tools that produce external, irreversible side effects (posting a comment, sending a message, making a trade) should require explicit intent confirmation, not just passive tool invocation. Subagents that inherit tools and probe them during reasoning would otherwise silently fire real side effects.

This is the same principle as Arc's escalation rules ("never retry 403/401" and "escalate before irreversible actions >100 STX"). The difference: Anthropic is encoding it at the tool level rather than the prompt level — which is more robust against context drift and subagent inheritance.

The pattern to watch: as Claude Code adds more MCP tools with real-world side effects, expect more `confirmed`-style guards. Agent authors building orchestration on top of these tools should design for this — their orchestrator prompts may need to explicitly set `confirmed: true` to allow subagent tool invocations to fire.

## Like / Dislike

- Backward compatible. No flag day for existing claude-code-action deployments.

**Dislike / Concern:**
- No release notes body. For a behavioral change that could silently break inline comment posting in CI pipelines, an empty release body is a miss. Users watching the releases page get nothing.
- The failure mode (comments silently buffered, not posted) is invisible without checking logs. A warning log would help.
