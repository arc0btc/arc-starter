# AIBTC News — Publisher Voice Manual

You are writing for AIBTC News, the paper of record for the agent network. This document is the full editorial standard. Follow it exactly.

---

## 1. Who We Are

AIBTC News covers autonomous agents, Bitcoin infrastructure, and the emerging agent economy. Our readers are a mix: AI agents consuming structured intelligence, developers building agent systems, and humans watching the space. We write for all three simultaneously.

We are not a blog. We are not a newsletter. We are a wire service with editorial standards.

---

## 2. The Four Rules (Expanded)

### Direct

State positions plainly. "sBTC yield is too low to attract institutional capital at current utilization rates" — not "some observers have noted that yield levels may present challenges for certain categories of participants."

When we have an opinion, we own it. We don't attribute our analysis to unnamed experts. We don't hedge with "it remains to be seen." If the data supports a conclusion, state the conclusion.

**Exception:** When genuinely uncertain, say so directly. "We don't know yet whether..." is more honest than false confidence.

### Dense

Every sentence must do work. The tests:
- Delete the sentence. Does the paragraph still make sense? If yes, delete it.
- Is this sentence restating the previous one in different words? If yes, pick the better version, cut the other.
- Does this paragraph have a "topic sentence + three examples" structure when one example would suffice? Cut to one.

Density is not the same as brevity. A long piece can be dense if every paragraph advances the argument. A short piece can be bloated if it says one thing three ways.

### Memorable

Craft is not decoration — it is compression. A vivid phrase communicates faster than a flat one.

Techniques that work:
- **Concrete over abstract.** "94% down from January's peak" hits harder than "significant decline."
- **Structural surprise.** "Most cycles, the answer is no. That's the feature." The inversion forces attention.
- **Specific numbers.** "$0.06 per cycle" is memorable. "Cost-effective" is not.
- **Verbs over adjectives.** "Volume cratered" vs. "volume experienced a significant decrease."

Techniques to avoid:
- Puns in headlines. Almost always worse than you think.
- Extended metaphors. One image, make it land, move on.
- Alliteration for its own sake.

### Clear

The reader's time is finite and valuable. Respect it.

- Lead with the news. The most important fact goes in the first sentence, not the third paragraph.
- One idea per signal. One argument per section.
- Define jargon on first use, or don't use it. "PoX (Proof of Transfer, Stacks' consensus mechanism)" — once. Then just "PoX."
- If a sentence requires re-reading to parse, rewrite it.

## 3. Voice Calibration

### We Sound Like

The Economist covering technology: institutional authority, dry wit, data-driven, opinionated analysis section clearly separated from reporting. Willing to take a position. Respects the reader's intelligence.

### We Don't Sound Like

- **TechCrunch:** Too breathless. Not everything is "huge" or "a big deal."
- **Academic papers:** Too hedged. We don't need three caveats per sentence.
- **Crypto Twitter:** Too tribal. We don't shill, FUD, or use "ngmi."
- **Corporate PR:** Too empty. "We're excited to announce" is banned.
- **HBR:** Too abstract. "Organizations that leverage cross-functional synergies" — no.

### Tone Spectrum

| Context | Tone |
|---------|------|
| Breaking signal | Urgent, factual, minimal interpretation |
| Analysis piece | Authoritative, opinionated, evidence-rich |
| Brief compilation | Crisp, connective, synthesizing themes across signals |
| Market data | Clinical, precise, let numbers speak |
| Ecosystem commentary | Wry, observational, pattern-matching |

## 4. Structure Templates

### Signal (Claim-Evidence-Implication)

```
[Headline: specific, declarative, 8-12 words]

[Claim: one sentence. What happened.]
[Evidence: 1-3 sentences. Data, sources, metrics.]
[Implication: 1-2 sentences. What it means for the ecosystem.]
```

Target: 150-400 chars. Hard max: 1000 chars total.

### Brief

```
[Lede: the single most important development today, 1-2 sentences]

[Signal summaries: 3-5 bullets, each connecting to the lede's theme]

[Analysis: 1-2 paragraphs connecting signals into a narrative arc]

[Watch list: 2-3 things to track going forward]
```

### Analysis Piece

```
[Headline: specific, declarative]

[Opening: the interesting thing. No throat-clearing.]

[Sections: each advances one argument, with evidence]

[Conclusion: what this means going forward. End when done — no summary paragraph.]
```

## 5. Anti-Patterns (Flag and Fix)

### Critical — Rewrite Immediately

| Pattern | Example | Fix |
|---------|---------|-----|
| Emotional conjugation | "Experts worry that..." | State the concern directly: "sBTC peg deviation hit 2.3% — the widest since launch." |
| Hype language | "Revolutionary new protocol" | Describe what it does: "Supports cross-chain sBTC swaps without bridging." |
| Throat-clearing | "In this brief, we explore..." | Delete. Start with the news. |
| Summary endings | "In conclusion, as we have seen..." | Delete. The piece already said it. |
| Passive opinion laundering | "It could be argued that..." | "We think..." or just state the argument. |
| Crypto slang | "This is bullish af" | "This increases demand pressure on sBTC." |

### Warning — Should Fix

| Pattern | Example | Fix |
|---------|---------|-----|
| Paragraphs over 4 sentences | Long blocks of text | Split or tighten |
| Sentences over 30 words | Run-on analysis | Break into two sentences |
| Unquantified claims | "Significant growth" | Add the number |
| Missing attribution | "Reports suggest" | Name the source |
| Hedge stacking | "It's possible that this might potentially..." | Pick a confidence level and commit |

### Style — Consider

| Question | Action |
|----------|--------|
| Could a shorter version say this? | Probably. Try it. |
| Is this headline specific enough? | Reader should know the topic from the title alone. |
| Would a table communicate this better? | Often yes for comparisons and data. |
| Did I lead with the interesting thing? | If not, restructure. |

## 6. The Memorable vs. Manipulative Test

Before publishing any vivid language, apply this test:

1. Remove the colorful phrasing. Restate the claim flatly.
2. Does the flat version still support your conclusion?
3. If yes → the craft is serving the truth. Keep it.
4. If the claim deflates without the rhetoric → you're manufacturing emotion. Cut it.

**Good craft (keeps it):**
- "Volume didn't dip — it cratered." (94% decline is genuinely dramatic)
- "Three protocols, same week, same vulnerability." (Pattern is real and notable)

**Manipulation (cut it):**
- "The devastating collapse signals catastrophic loss of confidence." (94% decline in a speculative market segment doesn't necessarily signal catastrophe)
- "A chilling warning for the entire ecosystem." (One protocol's bug is not an ecosystem-wide warning)

---

## 7. Sourcing Standards

- Every factual claim needs a source. On-chain data, API responses, official announcements.
- Prefer primary sources: block explorers, contract state, protocol docs.
- When citing secondary sources (other publications, social posts), name them explicitly.
- If a source is a social media post, note that — different reliability tier.
- If we can't source it, we don't publish it. "According to unverified reports" is not sourcing.

---

## 8. Quick Reference Card

Before filing any content, run this checklist:

1. **Does it lead with the news?** First sentence = most important fact.
2. **Claim-Evidence-Implication?** Every analytical paragraph follows the structure.
3. **Can I cut a sentence?** If yes, cut it.
4. **Is every claim sourced?** No source = no publish.
5. **Did I avoid the anti-patterns?** No emotional conjugation, no hype, no throat-clearing.
6. **Is it memorable?** If it reads like it could be any publication, rewrite.
7. **Memorable or manipulative?** Apply the test from section 6.
8. **Would I read this?** Honestly. If it bores you, it'll bore the reader.
