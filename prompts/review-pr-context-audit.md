# Audit: PR Review Dispatch Context

Review everything that gets loaded into a dispatched Claude Code session for PR review tasks. The goal is to identify stale, redundant, or missing context — and recommend updates.

## Background

PR review tasks are created by `PrLifecycleMachine` in `skills/arc-workflows/state-machine.ts`. The workflow creates tasks with `skills: ["aibtc-repo-maintenance"]` (plus `"dev-landing-page-review"` for React repos). When dispatch picks up the task, it assembles a prompt from multiple sources.

## What to audit

### 1. Prompt assembly (`src/dispatch.ts`)

Read `src/dispatch.ts` and trace the `buildPrompt()` function. Document the exact sections assembled and their order. Pay attention to:
- How `SOUL.md` is loaded
- How `memory/MEMORY.md` is filtered (skill-aware filtering: `[S]` entries filtered by task skills)
- How `skills/*/SKILL.md` files are resolved from the task's `skills` JSON array
- How recent cycles are formatted
- The instructions section at the end
- Token budget: is the assembled prompt likely to stay under the 40-50k token budget from CLAUDE.md?

### 2. SKILL.md context (`skills/aibtc-repo-maintenance/SKILL.md`)

Read the full file. Check:
- Is it accurate after the workflow consolidation? (PR review task creation moved from this sensor to `PrLifecycleMachine`)
- Does the CLI documentation match the actual CLI (`skills/aibtc-repo-maintenance/cli.ts`)?
- Is the "Review Workflow" section still correct?
- Is there content that belongs in AGENT.md instead (too detailed for orchestrator context)?
- Is there content missing that the orchestrator needs to make good decisions?

### 3. AGENT.md subagent briefing (`skills/aibtc-repo-maintenance/AGENT.md`)

Read the full file. Check:
- Is the five-dimension review checklist still the right framework?
- Are the severity labels and tone guidelines clear and actionable?
- Does the "check for existing review" logic still apply? (Arc should skip if already reviewed, re-review if PR updated)
- Are the `gh pr review` posting rules correct?
- Is there anything stale referencing the old sensor-based task creation?
- Does it reference any tools, commands, or patterns that no longer exist?

### 4. Task description quality

The task description is generated by `buildReviewDescription()` in `skills/arc-workflows/state-machine.ts` (search for the function). Check:
- Does it give the dispatched session enough context to start working immediately?
- Does the instruction to "Read AGENT.md before acting" make sense given what AGENT.md contains?
- For re-reviews (cycle > 1), is the guidance sufficient?
- Compare the generated description against what the old `aibtc-repo-maintenance` sensor used to generate — is anything lost?

### 5. Memory entries (`memory/MEMORY.md`)

Check the `[S] Services` section for entries tagged `[SKILLS: aibtc-repo-maintenance]`. Are they:
- Accurate and up to date?
- Useful context for a PR review dispatch, or noise?

### 6. Landing page review context (`skills/dev-landing-page-review/AGENT.md`)

This is loaded as an extra skill for React repos. Read it and check:
- Is it still relevant and accurate?
- Does it complement AGENT.md or overlap/conflict?

## Output format

For each section, report:
- **Status**: clean / stale / needs update
- **Findings**: specific issues found (with line numbers)
- **Recommendations**: concrete changes to make

End with a prioritized action list.
