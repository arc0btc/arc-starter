## Phase 4c: Memory Write-Back (Task #6604)

**Commit:** a5b82b3b on fix/failure-triage-error-patterns

**Implemented:**
1. **src/memory-writeback.ts** — New module with pattern-based learning extraction from task result_detail
   - 6 extraction patterns: root-cause/symptom-fix (incidents), patterns, learnings/lessons, prevention notes, architecture decisions
   - Jaccard word-overlap dedup (0.6 threshold) against existing FTS5 memory before writing
   - Auto-TTL: incidents=90d, patterns/prevention/decisions=180-365d, learnings=365d
   - Auto-tags: source:task:ID, priority:N, skill:name
   - Domain consolidation trigger when count > 75
   - Max 5 learnings per task to prevent flooding
2. **dispatch.ts** — Post-cycle hook after tagMemoryCostByTask, wrapped in try-catch (non-fatal)
   - Only fires for P1-4 tasks (high-value work worth remembering)
   - Logs stored count, dedup count, and consolidation status
3. **CLI** — `arc memory write-back --task ID [--dry-run]` for manual extraction and review

**Design decisions:**
- Inline in dispatch (not sensor) — learnings are freshest immediately after completion
- Pattern-based not LLM-based — zero additional cost, deterministic, fast
- Non-fatal wrapping — write-back failure never blocks dispatch
- Skips arc memory add matches to avoid double-writing entries the LLM already stored via CLI
