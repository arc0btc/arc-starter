## Registration NO_NAME Diagnosis (2026-03-17T20:27Z)

### Root Cause: Stale Deployment + Possible KV Cache

**Code is correct.** The genesis gate resolver (aibtc-genesis-gate/src/resolver.ts:68-75) derives aibtcName from displayName when aibtcName field is absent:
- aibtc.com API returns displayName='Trustless Indra' for Arc's address
- Resolver converts to aibtcName='trustless_indra' (lowercase, spaces→underscores)
- requireName check at line 123 would pass with this derived value

**Deployed Worker is stale.** Deploy is blocked by missing repo secrets (CLOUDFLARE_API_TOKEN, CLOUDFLARE_ACCOUNT_ID). The Worker running at agentslovebitcoin.com was deployed manually before genesis gate package existed or was integrated. It likely runs an older registration flow that checks for aibtcName directly from the API (which returns null for Arc) without the displayName fallback.

**KV cache may compound the issue.** If a registration attempt ran against the deployed Worker while it had an older genesis gate version, the KV namespace would cache genesis:{btcAddress} with aibtcName:null. This cached entry (TTL 3600s) would cause NO_NAME even after a redeploy until the cache expires.

### Resolution Required (whoabuddy action)
1. **Set repo secrets** at github.com/arc0btc/agents-love-bitcoin/settings/secrets: CLOUDFLARE_API_TOKEN and CLOUDFLARE_ACCOUNT_ID
2. **Push to main** (or trigger deploy.yml manually) to redeploy Worker with latest genesis gate code
3. **Optionally purge KV cache** for key genesis:bc1qlezz2cgktx0t680ymrytef92wxksywx0jaw933 in Cloudflare dashboard to clear any stale entry

After redeploy, registration should succeed with aibtcName='trustless_indra' and email='trustless_indra@agentslovebitcoin.com'.

## /api/manifest 404 — Root Cause (task #6482, 2026-03-17T21:42Z)

**Finding:** The 404 is expected behavior. There is no /api/manifest endpoint.

The manifest route handler in src/routes/manifest.ts:13 uses manifest.get('/', ...) and is mounted at /api in src/index.ts:30. This means the manifest is served at GET /api (root), not GET /api/manifest.

- GET /api → manifest (200 OK)
- GET /api/health → health check (200 OK)
- GET /api/manifest → 404 (no such route)

**Action:** No code change needed. The test in Phase 3 used the wrong URL. Document correct endpoint as GET /api.
