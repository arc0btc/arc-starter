## BIP-322 Fix for ALB Registration

### Problem
verifyBtcSignature() in btc-verify.ts tried BIP-137 first for ALL signatures (including bc1q addresses). If a wallet produced a 65-byte signature with wrong header bytes (e.g., P2PKH range 27-34 instead of P2WPKH 39-42), BIP-137 would fail and the code returned an error WITHOUT falling through to BIP-322.

### Fix Applied (local files in github/arc0btc/agents-love-bitcoin/)

**src/services/btc-verify.ts:**
- Reordered verification: BIP-322 first for bc1q/tb1q addresses, BIP-137 as fallback
- For non-segwit addresses: BIP-137 first (unchanged behavior)
- For segwit (bc1q): BIP-322 → BIP-322 legacy hash → BIP-137 fallback
- Updated file header and JSDoc comments

**src/routes/register.ts:**
- Updated comments from 'BIP-137+SIP-018' to 'BIP-322+SIP-018'

### Genesis-gate alignment
aibtc-genesis-gate does NO signature verification — it only resolves genesis status via aibtc.com API. No changes needed there.

### Key code change in verifyBtcSignature():
```typescript
const isSegwit = btcAddress.startsWith('bc1q') || btcAddress.startsWith('tb1q');

// Non-segwit: BIP-137 first
if (\!isSegwit && isBip137Signature(sigBytes)) { ... }

// BIP-322 P2WPKH path — primary for bc1q
if (bip322VerifyP2WPKHCore(signature, btcAddress, bip322TaggedHash(message))) return { valid: true };
if (bip322VerifyP2WPKHCore(signature, btcAddress, bip322TaggedHashLegacy(message))) return { valid: true };

// BIP-137 fallback for bc1q (wallets still producing compact sigs)
if (isSegwit && isBip137Signature(sigBytes)) { ... }
```

## x402 Payment-Required Header Verification

### Result: Header format is CORRECT — no fix needed

Verified ALB implementation against:
1. **coinbase/x402** (canonical spec) — `encodePaymentRequiredHeader()` = `safeBase64Encode(JSON.stringify(paymentRequired))`
2. **aibtcdev/x402-sponsor-relay** (actual relay at x402-relay.aibtc.com) — `/settle` endpoint expects `{ paymentPayload, paymentRequirements }`

### Header format details:
- **Header name**: `payment-required` (lowercase, HTTP headers are case-insensitive)
- **Header value**: base64-encoded JSON of PaymentRequiredBodyV2
- **PaymentRequiredBodyV2 schema**: `{ x402Version: 2, resource: { url, description, mimeType }, accepts: [PaymentRequirementsV2] }`
- This matches the x402 V2 spec exactly

### Settle request format:
ALB sends to `/settle`:
```json
{
  "paymentPayload": { "x402Version": 2, "payload": { "transaction": "<hex>" }, ...extensions },
  "paymentRequirements": { "scheme": "exact", "network": "<CAIP-2>", "amount": "<string>", "asset": "<sip010:...>", "payTo": "<addr>", "maxTimeoutSeconds": 60 }
}
```
This matches the relay's `validateV2Request()` exactly.

### Minor optional field differences (non-breaking):
- ALB omits `extra: Record<string, unknown>` on PaymentRequirementsV2 (optional in spec)
- ALB omits `extensions?: Record<string, unknown>` on PaymentRequiredBodyV2 (optional in spec)
- ALB omits `error?: string` on PaymentRequiredBodyV2 (optional, used for error responses)

### Conclusion:
whoabuddy suspected the header might be 'just a string' but the x402 V2 spec requires base64-encoded JSON, and ALB implements this correctly. No changes needed.
