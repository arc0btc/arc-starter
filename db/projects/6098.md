## Overview

Two new active state machines that replace whoabuddy manual /aibtc-org-review + repo review loop. Both use a shared `orgTier` context field ("managed" vs "collaborative") so aibtcdev can graduate to full autonomy by flipping one value.

### Relationship to Existing Infrastructure

- **Replaces**: Manual whoabuddy review loop, ad-hoc sensor→task pattern
- **Extends**: Existing `PrLifecycleMachine` (passive tracking only → active task creation)
- **Reuses**: `aibtc-repo-maintenance` skill for actual review execution, `github-issue-monitor` sensor for detection
- **Does NOT replace**: `github-mentions` sensor (notification routing), `arc0btc-pr-review` (paid service)

---

## 1. Issue Triage Machine (`github-issue-triage`)

### States

```
detected → searching → assessed → tagged → advised → closed
                                                  ↘ escalated
```

| State | Action | Task Created | Priority |
|-------|--------|-------------|----------|
| `detected` | Issue found by sensor | None (auto-transition to searching) | — |
| `searching` | Find related issues across repos | `Search related issues for {owner}/{repo}#{number}` | P7 (Sonnet) |
| `assessed` | Check code state: recent commits, open PRs, CI status | `Assess code state for {owner}/{repo}#{number}` | P7 (Sonnet) |
| `tagged` | Apply labels based on assessment | `Tag and classify {owner}/{repo}#{number}` | P9 (Haiku) |
| `advised` | Post repo-specific advice comment | `Post triage advice on {owner}/{repo}#{number}` | P6 (Sonnet) |
| `escalated` | Issue needs human attention (from any state) | None (terminal) | — |
| `closed` | Triage complete | None (terminal) | — |

### Transitions

```
detected    → searching
searching   → assessed, escalated
assessed    → tagged, escalated
tagged      → advised, escalated
advised     → closed
escalated   (terminal)
closed      (terminal)
```

### Context Schema

```typescript
interface IssueTriageContext {
  owner: string;
  repo: string;
  number: number;
  title: string;
  url: string;
  author: string;
  orgTier: "managed" | "collaborative";  // ← the autonomy gate
  labels?: string[];
  relatedIssues?: string[];       // "owner/repo#N" references found
  relatedPRs?: string[];          // PRs that address this issue
  ciStatus?: "passing" | "failing" | "none";
  severity?: "critical" | "moderate" | "low";
  triageNote?: string;            // Arc advice posted as comment
  lastChecked?: string;
}
```

### Instance Key

`issue-triage:{owner}/{repo}#{number}`

### Behavior by orgTier

| Action | managed (arc0btc) | collaborative (aibtcdev) |
|--------|-------------------|--------------------------|
| Label | Apply directly | Apply directly |
| Comment | Post advice | Post advice |
| Close stale | Close directly | Never close — flag for whoabuddy |
| Fix | Open PR via fix workflow | Open PR, whoabuddy merges |
| Assign | Self-assign | Do not assign |

### Sensor Integration

The existing `github-issue-monitor` sensor creates the workflow instance in `detected` state. The meta-sensor (arc-workflows) evaluates and creates tasks for each active state.

### Auto-Advance Logic

Each task, on completion, transitions the workflow to the next state. The meta-sensor picks up the new state and creates the next task. This means:
- `searching` task completes → stores `relatedIssues` in context → transitions to `assessed`
- `assessed` task completes → stores `ciStatus`, `severity` → transitions to `tagged`
- `tagged` task completes → stores `labels` → transitions to `advised`
- `advised` task completes → transitions to `closed`

Any task can transition to `escalated` if it determines human attention is needed.

## 2. PR Review Machine (`github-pr-review`)

### States

```
opened → reviewing → simplifying → advising → ready
                                            ↘ changes-requested
ready → merging → merged
              ↘ merge-blocked
changes-requested → reviewing (loop back)
```

| State | Action | Task Created | Priority |
|-------|--------|-------------|----------|
| `opened` | PR detected by sensor | None (auto-transition to reviewing) | — |
| `reviewing` | File-change review (diff analysis, correctness, security) | `Review PR {owner}/{repo}#{number}: {title}` | P4 managed / P5 collaborative (Sonnet) |
| `simplifying` | Run /simplify pass on changed files | `Simplify pass on {owner}/{repo}#{number}` | P5 (Sonnet) |
| `advising` | Post repo-context review comment | `Post review on {owner}/{repo}#{number}` | P5 (Sonnet) |
| `changes-requested` | Waiting for author to push fixes | None (sensor watches for new commits) | — |
| `ready` | Review complete, PR is approvable | Gate check (see below) | — |
| `merging` | Merge the PR | `Merge PR {owner}/{repo}#{number}` | P8 (Haiku) |
| `merge-blocked` | Waiting for whoabuddy approval | None (sensor watches for approval) | — |
| `merged` | Terminal | None | — |
| `closed` | Terminal (PR closed without merge) | None | — |

### Transitions

```
opened              → reviewing
reviewing           → simplifying, changes-requested, closed
simplifying         → advising, changes-requested
advising            → ready, changes-requested
changes-requested   → reviewing (re-review on new commits)
ready               → merging, merge-blocked
merging             → merged, closed
merge-blocked       → merging (when approval detected)
merged              (terminal)
closed              (terminal)
```

### Context Schema

```typescript
interface PrReviewContext {
  owner: string;
  repo: string;
  number: number;
  title: string;
  url: string;
  author: string;
  orgTier: "managed" | "collaborative";  // ← the autonomy gate
  requireApproval: boolean;               // ← explicit merge gate
  approver?: string;                      // who needs to approve (e.g. "whoabuddy")
  reviewers?: string[];
  reviewRound: number;                    // tracks re-review cycles
  filesChanged?: string[];
  simplifyRan?: boolean;
  reviewPosted?: boolean;
  ciStatus?: "passing" | "failing" | "pending" | "none";
  reviewDecision?: "approve" | "request-changes";
  approvalDetected?: boolean;
  mergeMethod?: "squash" | "merge" | "rebase";
  lastChecked?: string;
  closingIssues?: number[];               // issues this PR closes
}
```

### Instance Key

`pr-review:{owner}/{repo}#{number}`

(Matches existing `pr-review:` canonical key used by github-mentions and aibtc-repo-maintenance for cross-sensor dedup.)

### The Merge Gate

The `ready` state action function checks `requireApproval`:

```typescript
ready: {
  on: { merge: "merging", block: "merge-blocked" },
  action: (ctx) => {
    if (ctx.ciStatus === "failing") return { type: "noop" }; // wait for CI
    if (ctx.requireApproval && \!ctx.approvalDetected) {
      return { type: "transition", nextState: "merge-blocked" };
    }
    return { type: "transition", nextState: "merging" };
  },
}
```

### Behavior by orgTier

| Action | managed (arc0btc) | collaborative (aibtcdev) |
|--------|-------------------|--------------------------|
| Review | Full review + /simplify | Full review + /simplify |
| Post review | `gh pr review --approve` or `--request-changes` | `gh pr review --comment` (never approve/reject) |
| Merge | Auto-merge when CI passes | Wait for whoabuddy approval |
| Close | Can close abandoned PRs | Never close |
| requireApproval default | `false` | `true` (approver: "whoabuddy") |

### Graduating aibtcdev

To give aibtcdev full autonomy:
1. Set `requireApproval: false` in workflow context for new aibtcdev PRs
2. Change review comment to `--approve` / `--request-changes`
3. No template changes needed — same machine, different context

### Sensor Integration

**PR detection**: Existing `aibtc-repo-maintenance` sensor and `arc-workflows` PR sync already detect PRs. Modify to create `github-pr-review` workflow instances instead of (or in addition to) `pr-lifecycle` instances.

**State sync**: The existing `mapPRStateToWorkflowState()` in `arc-workflows/sensor.ts` handles GitHub→workflow state mapping. Extend it to detect:
- New commits after `changes-requested` → transition back to `reviewing` (increment `reviewRound`)
- whoabuddy approval on `merge-blocked` PRs → set `approvalDetected: true`, transition to `merging`

### Re-review Loop

When `changes-requested` detects new commits:
1. Sensor sees commit SHA changed since last check
2. Transitions to `reviewing` with `reviewRound++`
3. New review task created — reviews only the new changes (delta review)
4. Cycle continues: reviewing → simplifying → advising → ready

## 5. Key Design Decisions

1. **Single template per machine, orgTier in context** — not separate machines per org. Graduating aibtcdev = changing context defaults, not swapping templates.

2. **requireApproval as explicit boolean** — clearer than inferring from orgTier. Allows per-repo override (e.g., arc0btc repo that needs approval for some reason).

3. **Re-review loop** — `changes-requested → reviewing` with `reviewRound` counter. Delta reviews on subsequent rounds (only review new commits, not full diff).

4. **Merge via task, not sensor** — merging is an action, not a state detection. The `merging` state creates a P8 Haiku task that runs `gh pr merge`.

5. **Issue triage is linear** — no loops. If triage reveals the issue needs a PR, that spawns a separate PR workflow (not a loop back within issue triage).

6. **Instance key alignment** — PR review uses `pr-review:{owner}/{repo}#{number}` matching existing dedup keys across github-mentions and aibtc-repo-maintenance sensors.

7. **Coalesce review + simplify + advise into workflow** — currently these happen ad-hoc in a single dispatch. The workflow breaks them into stages so each can be independently tracked, retried, and attributed for cost.

# Gitea↔GitHub Contribution Flow Design

**Task 6101** | P4 Opus | Source: whoabuddy email 2026-03-16

---

## Core Principle: One Canonical Direction Per Repo

No bidirectional sync. Every repo has exactly one canonical source. The other side gets a one-way mirror or port.

---

## Repo Classification

| Canonical | Repos | Why |
|-----------|-------|-----|
| **GitHub** | aibtcdev/* (landing-page, skills, x402-api, aibtc-mcp-server, agent-news) | Community-facing, external contributors PR here, existing CI/CD |
| **Gitea** | arc-starter, arc0btc/* (arc0me-site, etc.), new fleet-internal repos | Fleet dev hub — agents push here directly without GitHub gate |

**Rule:** External humans contribute → GitHub-canonical. Fleet-only → Gitea-canonical.

---

## Flow 1: Gitea-Canonical → GitHub (Fleet → Public)

1. Fleet agents push branches/PRs to Gitea directly (no GitHub creds needed)
2. Arc reviews on Gitea using Gitea API
3. After merge to main, bridge sensor on Arc pushes main to GitHub
4. GitHub is read-only mirror for gitea-canonical repos

Bridge sensor (Arc-only, every 5 min):
- For each gitea-canonical repo: fetch gitea main, fetch github main
- If gitea ahead: git push github main
- Simple push, no merge logic — gitea main is truth

## Flow 2: GitHub-Canonical → Fleet (Public → Fleet)

1. Gitea mirrors GitHub repos (built-in mirror, pulls every 10 min, read-only)
2. External PRs reviewed on GitHub via existing github-pr-review machine
3. Fleet contributions: agent → fleet-handoff to Arc with patch/bundle → Arc opens GitHub PR
4. No writes to Gitea mirrors (prevents divergence)

## Flow 3: Fleet Agent Development (arc-starter)

1. Agent pushes branch to Gitea
2. Opens Gitea PR
3. Arc reviews via Gitea API
4. Merge to Gitea main
