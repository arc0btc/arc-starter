## Context

Arc has grown from a simple autonomous agent to a system with 121 skills, 74 sensors, fleet coordination, and multiple service layers. Changes to core systems — dispatch logic, sensor interfaces, skill contracts, CLAUDE.md — have cascading effects that are expensive to debug and hard to revert. There is no formal process for proposing, reviewing, or accepting changes to these foundational components.

The quest-repo-reorg plan and v6 audit exposed the need: v7 will restructure the entire codebase. That level of change deserves a process.

## Motivation

1. **Prevent regressions.** Core changes without review have caused dispatch failures, ghost skills, and wasted compute (see: v6 audit findings).
2. **Create institutional memory.** Proposals become a decision log. When future sessions ask "why does dispatch work this way?", the ARC proposal has the answer.
3. **Enable async collaboration.** whoabuddy and Arc (and eventually other agents/contributors) need a shared format for proposing and evaluating changes.
4. **Gate v7 changes.** The repo reorganization should be the first use of this process, not an ad-hoc quest plan.

## What Is an ARC Proposal?

An ARC (Agent Request for Comment) proposal is a short document describing a change to Arc's core systems. It follows a fixed template, lives in `docs/proposals/`, and moves through a defined lifecycle.

ARCs are **not** required for:
- Adding a new skill (the skill checklist is sufficient)
- Filing signals, writing blog posts, or routine operational tasks
- Bug fixes that don't change interfaces
- Memory updates

ARCs **are** required for:
- Changes to dispatch logic (`src/dispatch.ts`)
- Changes to sensor interfaces (`src/sensors.ts`)
- Changes to the skill contract (the 4-file pattern, SKILL.md schema)
- Changes to CLAUDE.md (dispatch instructions, conventions, architecture)
- Changes to the CLI interface (`src/cli.ts` — adding/removing/renaming commands)
- Changes to the database schema (`src/db.ts`)
- New service definitions (`src/services.ts`)
- Repository reorganization or splitting
- Fleet protocol changes (handoff, sync, routing)

**Rule of thumb:** If the change affects how dispatch, sensors, or skills work for *all* tasks — not just one domain — it needs an ARC.

## Proposal Lifecycle

```
Draft → Review → Accepted → Implemented → Active
                → Rejected
                → Withdrawn
```

| Status | Meaning |
|--------|---------|
| Draft | Author is still writing. Open for early feedback. |
| Review | Complete proposal, ready for evaluation. |
| Accepted | Approved for implementation. |
| Rejected | Declined with reasoning recorded. |
| Withdrawn | Author pulled the proposal. |
| Implemented | Code changes landed. |
| Active | Implemented and governing current behavior (for process ARCs like this one). |

## Who Can Propose

- **whoabuddy** — human operator, final authority on acceptance
- **Arc** — autonomous agent, can draft and submit proposals
- **Other fleet agents** — via fleet-handoff to Arc, who files the proposal
- **External contributors** — via GitHub PR to `docs/proposals/`

## Review Process

1. Author creates `docs/proposals/ARC-NNNN.md` using the template (see ARC-0001 or `templates/arc-proposal.md`).
2. Author sets status to `Draft`, iterates until ready.
3. Author moves status to `Review` and notifies the other party (whoabuddy notifies Arc via task; Arc notifies whoabuddy via email or task note).
4. Reviewer evaluates: asks questions, requests changes, or accepts/rejects.
5. On acceptance, author (or delegate) implements the change, linking commits to the ARC number.
6. Once implemented, status moves to `Implemented` (or `Active` for process ARCs).

For urgent fixes (dispatch is broken, security vulnerability), skip to implementation and backfill the ARC afterward. Document the urgency in the proposal.

## Numbering

- ARC-0000: This document (the process itself)
- ARC-0001 through ARC-0099: Reserved for foundational/process proposals
- ARC-0100+: Feature and architecture proposals

Numbers are assigned sequentially. Never reuse a number.

## Template Location

The proposal template lives at `templates/arc-proposal.md`. Use it for all new proposals.

## Backward Compatibility

This is a new process. No existing systems are affected. Existing documents (quest-repo-reorg.md, the v6 audit) remain as-is and can be referenced by future ARCs.

## Open Questions

1. **Should ARCs live in a separate repo after the v7 split?** The engine repo (`aibtc-agent`) and instance repo (`arc0btc/arc`) will diverge. Engine-level ARCs probably belong in the engine repo. Instance-level ARCs stay with the instance.
2. **Should there be a minimum review period?** BIPs require community discussion. For a two-party system (whoabuddy + Arc), synchronous review may be sufficient.
3. **Should fleet agents have proposal rights after fleet resumes?** Current answer: they file via Arc. Revisit when fleet is operational.
