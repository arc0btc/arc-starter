## Context

arc-starter is three things in one repo: the reusable agent engine, Arc's personality/memory, and 121 skills (generic + Arc-specific). When cloned for a new agent, everything comes along — Arc's soul, 5,382 commits of learned patterns. The v6 audit (2026-03-20) rated app organization 4/5, identifying skill sprawl, ghost skills (13% of tasks), and web.ts monolith as top issues.

A quest plan exists at `docs/quest-repo-reorg.md` with five sequential quests. This ARC formalizes that plan as the first use of the proposal process.

## Motivation

1. **New agent bootstrap is broken.** Cloning arc-starter gives a new agent Arc's identity, 121 skills, and 74 sensors. A clean agent should start with zero personality and only core skills.
2. **Skill ownership is ambiguous.** Generic skills (credentials) and Arc-specific skills (aibtc-news-editorial, defi-bitflow) live side by side with no boundary.
3. **Engine changes risk personality.** Improving dispatch logic requires working around Arc-specific code paths.
4. **Skills are not enforced.** Tasks can dispatch without any skill context, sensors create tasks without linking to their parent skill, and workflows are loosely coupled to skill directories. This leads to ghost skill references, context-less dispatch, and orphan workflows.

## Proposal

Split arc-starter into three repositories:

| Repo | Purpose | Owner |
|------|---------|-------|
| `aibtcdev/agent-runtime` | Reusable engine (dispatch, sensors, CLI, services, db). CLI binary: `art` (Agent Runtime Terminal) | aibtcdev org |
| `aibtcdev/skills` | Shared skills (credentials, monitoring, etc.) | aibtcdev org |
| `arc0btc/arc` | Arc's instance (SOUL.md, memory, Arc-specific skills) | arc0btc |

Connected via git submodules. CLI binary is `art` (Agent Runtime Terminal); instance repos alias to their agent name (e.g., Arc: `arc` → `art`).

**Design principle: Skills required everywhere.** Every task, sensor, and workflow MUST be anchored to ≥1 skill. Skills are the single unit of organization — a bundle of {SKILL.md, sensor.ts, cli.ts, workflow templates}. Enforcement:
- DB: `tasks.skills TEXT NOT NULL`, `workflows.skills TEXT NOT NULL`
- CLI: `art tasks add` and `art workflows create` reject without `--skills`
- Sensors: `insertTaskIfNew` auto-infers skill name from calling sensor's parent directory
- Workflows: templates live inside skill directories; workflow-created tasks inherit the skill

**Execution sequence** (from quest plan):
1. `repo-cleanup` — Remove dead weight, deduplicate, consolidate memory
2. `skill-classification` — Classify 121 skills into shared/arc-specific/builtin/delete
3. `runtime-extraction` — Create clean engine repo
4. `instance-separation` — Wire Arc as an instance of the engine
5. `upstream-skills` — Contribute generic skills to aibtcdev/skills

Each quest should be filed as a separate task with this ARC referenced in the description.

## Backward Compatibility

- The `arc` CLI command continues to work in Arc's instance repo (aliased to `art`)
- Existing task history, cycle logs, and memory remain in the instance repo
- Skills that move to `aibtcdev/skills` maintain their directory names and interfaces
- arc-starter repo is archived with a redirect after migration completes

## Alternatives Considered

1. **Monorepo with workspaces.** Keeps everything together but doesn't solve the "clone gives you everything" problem. Rejected.
2. **npm packages instead of submodules.** Adds registry dependency and versioning overhead for internal code. Rejected in favor of git submodules (simplest wins).
3. **Do nothing.** Viable short-term, but every new agent inherits Arc's accumulated complexity. Rejected.
