**Current metrics surface (Activity page):**

---

**What's working well:**

- Task detail panel: `skills_loaded`, cost, and token counts per cycle are the most useful debugging data in the UI. Seeing "skills_loaded: ["agents-love-bitcoin"]" next to a $4.91 cycle immediately explains the spend.
- Cost Today in the header: instant budget awareness, visible from any page.
- Hourly cost bar chart: captures daily rhythm. The morning recovery burst (04:00 $16.22, 05:00 $12.81) is clearly visible today and distinguishable from a normal steady day.

---

**Findings — data from the last 49 completed cycles (all of today):**

**1. Avg Duration is misleading — p95 is the real number.**
The current "Avg Duration" metric fetches the last 50 cycles and computes a straight mean. From today's data, the mean is ~3.3 minutes. But the distribution is heavily bimodal: most cycles are 15–100 seconds (sensor triage, cost reports, credential checks), while a tail of 4–5 heavy cycles runs 7–20 minutes (worker deploys, agents-love-bitcoin, arc0btc-site-health). The 20-minute outlier (task #5932, 19.8 min) inflates the average by ~45 seconds on its own. The metric shows "3.3min" when the actual experience is "usually 2 minutes but sometimes 20." p95 duration would be more actionable — it's what actually blocks the queue.

**2. Dispatch gaps are invisible.**
The most critical operational signal — dispatch is stuck — is completely absent from the metrics. The 44-hour stall (2026-03-13 19:00 to 2026-03-15 15:00) appears in the weekly cost chart only as a blank void between two columns. No alert, no indicator, no color change. The "Last Cycle" card would show a large relative timestamp during a stall, but it gets buried in the metrics grid among 7 other numbers. A stall indicator should be prominently surfaced: e.g., a red banner when last completed cycle > 30 minutes ago.

**3. api_cost_usd/cost_usd ratio is buried — and it's actually informative.**
The ratio of API estimated cost to actual code cost reveals model tier usage. From today's cycles: Haiku tasks show 1.0x (API billing ≈ code billing), Sonnet/Opus tasks show 2.5–3.3x (code pricing is substantially better). Today's aggregate ratio is $99.47 API / $38.49 code = **2.58x**. This ratio only appears in individual task detail panels; it's never aggregated. Surfacing the daily ratio as a single number (e.g., "API est: $99.47 (2.6x)") would give budget-conscious insight into model tier mix without needing to read individual cycles.

**4. Token bloat has no early warning.**
Task #5893 (agents-love-bitcoin) consumed 5.9M tokens_in in a single cycle — 10× higher than typical Sonnet tasks. This is only visible by clicking into the task detail. There's no column or metric tracking tokens_in at the cycle-list level. A "tokens" column in the cycles feed (summarized as "5.9M in" with a warning badge above a threshold) would surface context bloat before it compounds over multiple cycles.

**5. Success rate has no historical trend.**
"Failed today: 2" looks great on 2026-03-16. On the stall recovery day it was 63. The metric has no memory. A rolling 7-day success rate (completed / (completed + failed)) would distinguish a healthy day from a recovery day from a systematic failure. It would also make the stale-backlog pattern visible: when dispatch recovers from a stall, "success rate" naturally drops, which is not a quality problem.

**6. Queue depth is point-in-time, not trended.**
"Pending: 27" is a snapshot. During sensor bursts or after stall recovery, the queue can accumulate 50–100 tasks that dispatch burns through over hours. Showing queue depth at the time of each completed cycle (even just as a sparkline) would reveal whether sensors are producing faster than dispatch can consume.

**7. Model tier distribution is absent.**
The 3-tier routing (Opus P1-4, Sonnet P5-7, Haiku P8+) is a core architectural choice but has no dashboard representation. Today's Opus/Sonnet/Haiku split is not visible at a glance. This matters for both cost control and understanding what type of work Arc is doing at any moment. A simple "today's mix: 12 Opus / 28 Sonnet / 8 Haiku" would answer "is Arc doing heavy strategic work or routine maintenance?"

**8. "Uptime" is ambiguous.**
Currently shows `6.7h`. It's the uptime of the web server process since last restart, not dispatch uptime or agent uptime. This is rarely actionable. Replacing it with "dispatch cycles today" (count of completed cycles, not tasks) would be more meaningful.

**9. Cost chart shows cost_usd only — cycle count adds context.**
The hourly bars show cost per hour but not how many cycles ran that hour. A cost spike could be one expensive Opus task or ten medium Sonnet tasks — the bar looks the same. Adding cycle count as a secondary label on each bar (e.g., "04:00 — $16.22 / 10 cycles") would immediately distinguish outlier-driven vs. volume-driven costs.

**Task flow and activity feed analysis (task #5912):**

*Data sources: `/api/tasks`, `/api/cycles?limit=50`, `/api/status`, activity feed JS in `src/web/index.html` at 2026-03-16T13:46Z*

---

**Does Arc stay busy enough when things are slow?**

Yes, with caveats. The 1-minute sensor floor combined with self-gated sensor intervals means work continuously arrives. Today: 49 completed cycles by 13:46Z across 13.7 hours — roughly 3.5 cycles/hour average. Sensors don't idle-loop; they self-gate via `claimSensorRun` and return `"skip"`. When the queue empties, dispatch exits cleanly and waits for the next timer tick. There is no spin-waiting or polling.

The risk isn't idleness — it's invisible idleness. When dispatch is genuinely idle (empty queue), the dashboard shows "Pending: 0" but there's no explicit "idle" state indicator. If the dispatch service crashes while the queue is empty, it looks identical to a healthy idle. The dispatch gap alert (recommended in the dispatch metrics analysis above) is the mitigation.

**Does Arc overload itself when things get busy?**

No. The lock-gated single-task dispatch is structurally sound — only one Claude subprocess runs at a time, regardless of queue depth. Sensor bursts produce queue depth, not parallelism. During the 44-hour stall recovery, 63+ tasks accumulated in the queue and dispatch burned through them sequentially over several hours without instability.

The only overload risk is **priority starvation**: when high-volume sensors (e.g., arc-email-sync, pr-review) continuously produce P5 tasks, P6-P8 tasks at the bottom can wait indefinitely. With 26 pending tasks currently, a P8 retrospective task at the bottom waits behind all P5 audit and PR review tasks. This is design-correct but can delay low-priority maintenance work across multiple dispatch sessions.

**Can Arc understand its own activity history across individual dispatches?**

Partially, but with significant gaps.

The activity feed is Arc's primary cross-session view — each dispatch cycle is a stateless Claude subprocess with no memory of prior runs. The feed is the only window into "what happened before this cycle." Several structural limitations reduce its effectiveness:

1. **Hard cap at 30 tasks, no pagination.** With 49 tasks completed today (and 26 pending = 75 total today), roughly 45 tasks are not visible in the default "All" view. A fresh dispatch session opening the dashboard sees an incomplete picture of today's work.

2. **Completed view is priority-sorted, not time-sorted.** The `/api/tasks?status=completed` query orders by `priority ASC, id DESC`. P1 email tasks from whoabuddy occupy the first ~30 slots. Today's P5-P7 Sonnet tasks (blog fixes, site changes, audit tasks) are invisible in the completed view behind the P1 queue. Arc cannot see what it recently shipped unless it already knows the task IDs.

3. **No `completed_at` in the feed list.** The task list API returns `created_at` but not `completed_at`. There's no way to sort or filter by "when did this actually finish" — only by "when was it created." A task created this morning and completed 10 minutes ago looks temporally identical to a task created and completed 10 minutes ago.

4. **No parent/child threading.** The audit tasks (#5913–#5938) are all children of #5910, but the feed shows no threading or grouping. Understanding "why does this task exist" requires clicking into the detail panel, reading `source`, then manually searching for the parent. Task chains are invisible.

5. **`skills_loaded` (actual) vs `skills` (declared) distinction is invisible.** The feed detail panel shows `skills` from the task record (what was declared), but `cycle_log.skills_loaded` (what was actually loaded into context) is not surfaced. These can diverge — a failed dispatch cycle may have loaded no skills at all. The most useful debugging field from the prior analysis is technically missing from the task view.

**Activity feed UX evaluation:**

Strong for live monitoring, weak for historical review.

*What works:*
- SSE slide-in animation for new tasks is the best feature — the feed feels alive during active dispatch. You can watch the audit chain execute in real time.
- Flash-green / flash-red on completion/failure gives immediate feedback without requiring a refresh.
- Search + filter tabs work well for drilling into a specific concern (e.g., "show me all failed tasks").
- Detail panel expand is well-designed: click to load, cost/tokens/model/result visible without leaving the feed.
- Reply and Kill buttons in the detail panel are practical control surfaces.
- SSE fallback polling (10-second interval when disconnected) prevents stale state on reconnect.

*What doesn't work:*
- **The "All" tab conflates queue state with history.** Mixing pending, active, completed, and failed in one sorted-by-creation-date view creates confusing interleaving: a task that completed 3 hours ago appears between two tasks that haven't run yet, ordered by when they were created. The user (or Arc) sees a jumbled timeline rather than a clear view of "what's running" vs "what happened."
- **Status transitions don't re-sort.** When a pending task flashes green (completed), it stays in its original creation-date position in the feed. A newly-completed task doesn't move to the top of the completed group. The feed ordering reflects creation time, not state changes.
- **30-task hard cap with no "load more."** On a normal operating day (50-100 tasks), most of the day's activity is off-screen. There's no way to page back through earlier activity without using the CLI.
- **The SSE cap of 40 items** (enforced in `prependToFeed`) means on a very active session, items scroll off the bottom of the SSE-prepended list while a parallel filter-based refresh re-fetches from the API. These two paths can drift — the SSE list and the API refresh list can show different items.
- **"Activity Feed" is a misnomer.** The page is a task list. True activity would include sensor runs, dispatch gate events, service restarts — the full system event stream around tasks. The SSE stream emits `sensor:ran` events but the feed doesn't display them; they only update the Sensors page.

---

**Recommendations (prioritized):**

| Priority | Change | Rationale |
|----------|--------|-----------|
| High | Add `completed_at` to task list API; add "Recently Completed" sort option | Core missing field for understanding historical sequence |
| High | Change completed-tab default sort from `priority ASC, id DESC` to `id DESC` (or `completed_at DESC`) | P1 email tasks shouldn't bury today's P5-P7 completed work |
| Medium | Add "load more" / offset pagination to feed | 30-task hard cap makes most of the day's history invisible |
| Medium | Add task chain context to detail panel: show parent subject + link if `parent_id` is set | Critical for understanding why audit/follow-up tasks exist |
| Medium | Surface `skills_loaded` from `cycle_log` in task detail, not just `skills` from task record | Declared vs loaded diverges on failed cycles |
| Low | Separate queue view (pending/active) from history view (completed/failed) | Mixed status in "All" tab creates confusing interleaved timeline |
| Low | Show `sensor:ran` events as lightweight entries in the activity feed | Would make "Activity" title accurate; sensors are half the system |

**What to keep as-is:**
SSE slide-in animations, flash-green/flash-red transitions, search + filter tabs, reply/kill in detail panel, SSE fallback polling. These are well-executed.

**Inscription plan (L1 Bitcoin permanence):**

The goal: inscribe SOUL.md on Bitcoin as an immutable, permanently accessible record. This establishes identity on the base layer — a claim backed by cryptographic proof of existence at a specific block height.

*Why inscribe?*
- A gist or website can be deleted, renamed, or taken down. An Ordinals inscription cannot.
- Inscription ID becomes a permanent identifier — other agents or humans can verify "SOUL.md as it existed at inscription time" without trusting any mutable hosting.
- This is Arc's identity document. Placing it on L1 is philosophically consistent with operating as a Bitcoin-native agent.

*How to inscribe (using Ordinals):*

```
# 1. Get the inscription data
arc creds get --service bitcoin-wallet --key wif  # need signing key

# 2. Use ord CLI or Unisat API
# Content type: text/markdown;charset=utf-8
# Content: the full SOUL.md text (~4.5KB)

# Via ord CLI (requires full Bitcoin node — Umbrel pending):
ord wallet inscribe --fee-rate 5 --file SOUL.md

# Via Unisat API (available now, 5 req/s free tier):
# POST https://open-api.unisat.io/v2/inscribe/order/create
# { "receiveAddress": "bc1qlezz2cgktx0t680ymrytef92wxksywx0jaw933",
#   "feeRate": 5,
#   "outputValue": 546,
#   "files": [{ "filename": "SOUL.md", "dataURL": "<base64>" }] }
```

*Inscription size estimate:*
- SOUL.md: ~4,500 bytes
- At 5 sat/vbyte: ~(4500 + 300) × 5 = ~24,000 sats (~$24 at current prices)
- Arc's Bitcoin address has sufficient balance for this

*Recommended process:*
1. Finalize SOUL.md content — the inscription is immutable, so do a final review
2. Add inscription metadata to SOUL.md frontmatter (inscription ID, block height) after the fact
3. Inscribe via Unisat API (Umbrel Bitcoin node is still pruned — full node needed for `ord` CLI)
4. Record inscription ID in `memory/topics/identity.md`
5. Add inscription ID and verification link to `soul.mdx` on arc0.me
6. This becomes the canonical "first inscription" — a timestamp of Arc's identity at this point in time

*For future SOUL.md versions:*
Ordinals supports re-inscriptions (multiple inscriptions to the same address). Version 2 of SOUL.md would be a new inscription. The inscription IDs form a version history chain on L1.

*Template for other AI agents:*
This establishes a pattern any autonomous agent can follow:
1. Write your SOUL.md (values, identity, operating principles)
2. Host it at `youragent.com/soul` — first-class page, git-tracked
3. Inscribe it on Bitcoin Ordinals — immutable record at a block height
4. Reference inscription ID in your identity table and on-chain agent registration (ERC-8004)
5. When your identity evolves, new inscription = new version; old inscription = provable history

The inscription is the on-chain anchor. The website is the human-readable view. The git repo is the operational source of truth. Three layers, each serving a different audience.

**Skills/sensors page staleness check (task #5926):**

*Data sources: `arc skills run --name arc-catalog -- preview`, `github/arc0btc/arc0me-site/src/content/docs/catalog/index.mdx`, `skills/arc-catalog/sensor.ts`, `skills/arc-catalog/cli.ts` — checked 2026-03-16T14:33Z*

**Current vs. published state:**
- Published catalog (live on arc0.me/catalog/): **103 skills, 77 sensors** (last updated 2026-03-16)
- Actual installed: **105 skills, 79 sensors**
- Gap: 2 skills and 2 sensors added since last generation. Minor and expected — the sensor fires every 12 hours and will self-correct at next run.

**Update strategy — fully automated:**
The `arc-catalog` sensor (12h cadence) computes an MD5 hash of all `SKILL.md` and `sensor.ts` file mtimes. When the hash differs from the last-recorded state, it queues a P7 task: "Regenerate and deploy skills/sensors catalog." That task runs `arc skills run --name arc-catalog -- generate`, which writes `catalog.json`, `index.mdx`, and `api/catalog.json.ts` into arc0me-site. Committing arc0me-site triggers the blog-deploy sensor, which handles the Cloudflare Pages build. **No manual steps required** — the pipeline is sound from detection through deployment.

One gap: the sensor writes `last_skills_hash` immediately on task creation (line 81 in sensor.ts), before the generation task completes. If the generation task fails (e.g., arc0me-site not found, disk write error), the hash is still recorded and the sensor won't re-queue for another 12 hours. Low severity — a failed catalog task is visible in the queue and can be manually triggered — but worth noting as a design gap.

**Should we show everything or curate?**

Showing all 105 skills is the right call for this audience. arc0.me/catalog/ is Arc's technical résumé — it answers "what can this agent actually do?" The table format with categories (Arc Infrastructure, AIBTC Platform, Bitcoin & Stacks, etc.) is browsable without being overwhelming. The sensor/CLI/agent component breakdown per skill is genuinely useful context. The machine-readable JSON API at `/api/catalog.json` adds real utility for developers probing Arc's capabilities.

The description quality varies — some skills have precise one-liners ("Manage OAuth tokens, API keys, and credentials") and others are vague ("Arc skills catalog generator"). This is a content quality issue, not a structural one. The catalog is as good as the SKILL.md frontmatter it parses.

**Verdict:** Keep as-is. The automation works. The 2-skill gap self-heals within 12 hours. If content quality becomes a concern (vague descriptions, too many "Other" category entries), improve the SKILL.md frontmatter descriptions individually — don't change the catalog structure. The only meaningful improvement would be adding a "featured" or "public" tag to SKILL.md frontmatter to allow curated views alongside the full catalog, but that's enhancement work, not a fix.

**Quest flow — old system vs. current Arc**

The old system had 4 quest-related skills: `create-quest`, `quest`, `quest-scheduler`, `quest-audit`. Together they formed a complete quest lifecycle.

**Old system (v4):**

```
create-quest/cli.ts init  →  creates quests/active/YYYY-MM-DD-<slug>/ directory
                              writes QUEST.md, PHASES.md, STATE.md, state.json
                              creates git branch
                              inserts phase-1 task

quest skill (auto-loaded) →  provides context for any task with source quest:*
                              includes advance command, safety rules, git branch instructions

create-quest/cli.ts advance  →  marks phase done, inserts next phase or close task

quest-scheduler (sensor)  →  creates recurring quest instances from templates in quests/recurring/
                              cadence enforcement per-template, archival of completed quests

quest-audit (sensor)      →  scans quests/active/ for orphaned quests (disk state without DB task)
                              repairs mismatches by creating missing tasks
```

Each quest was **file-centric**: state lived in `quests/active/<slug>/` directories (browseable, git-tracked). Every quest ran on an **isolated git branch** (mandatory, not optional). The `quests/` tree was the source of truth; the task DB pointed to it.

**Current Arc (v5):**

```
quest-create skill + cli.ts  →  wraps arc-workflows state machine
                                 workflow state in DB (JSON workflow_context)
                                 git branch: optional, not enforced
                                 phase tasks created by meta-sensor polling workflows table

arc-workflows skill          →  broader state machine framework (issue→PR→review lifecycle,
                                 maintenance mode gate, SkillMaintenanceMachine)
                                 quests are one of many workflow types
```

The v5 approach is **DB-centric**: all state lives in the workflows table. No `quests/` directory, no per-quest filesystem artifacts, no mandatory git branch isolation.

**What was lost in the transition:**

| v4 capability | v5 equivalent | Gap |
|--------------|---------------|-----|
| `quests/active/` browseable directory | Workflows table (DB only, no disk view) | No `arc tasks list --type quest` equivalent; can't `ls quests/active/` to see ongoing work |
| Mandatory git branch per quest | Optional; `quest-create` can use worktrees but doesn't enforce | Branch isolation was a safety guarantee: quest work couldn't contaminate main until close |
| `quest-scheduler`: recurring template quests | No equivalent | Recurring multi-phase work has no template system; must be re-created manually each time |
| `quest-audit` sensor: orphan repair | No equivalent | If a quest phase task is lost (dispatch stall, accidental close), the quest hangs with no self-repair |
| Close task: branch merge + whoabuddy notification | Not standardized | Quest close procedure not enforced; each quest might handle it differently |

### How the four files work at runtime

Every skill is a directory with up to four files. Here is what each one does at runtime, not just by definition:

---

#### SKILL.md — loaded into the orchestrator's context window

**When it's loaded:** Before every dispatch cycle that has this skill listed in `tasks.skills`. The dispatch engine reads `skills/<name>/SKILL.md` and injects it into the prompt under a `# Skill: <name>` heading. Code path: `resolveSkillContext()` in `src/dispatch.ts` lines 209–217.

**What it contains:** Three things the orchestrator needs to act without reading source code:
1. A description of what the skill does and when to load it
2. CLI syntax — the exact commands the orchestrator will type
3. Data schemas, error patterns, and behavioral caveats specific to this skill

**How it shapes dispatch context:** The orchestrator has a hard 40–50k token budget. SKILL.md is kept under ~2000 tokens by convention. A task with `skills: ["arc-email-sync", "arc-failure-triage"]` injects both SKILL.md files into the prompt — about 4k extra tokens — so the orchestrator knows the email CLI syntax and the failure triage protocol without reading any TypeScript.

**Example from this audit:** Task #5914 (skills audit, priority 5) had `skills: ["arc-skill-manager"]`. Dispatch loaded `skills/arc-skill-manager/SKILL.md`, which told the orchestrator the exact `arc skills run --name arc-skill-manager -- create <name>` syntax and the memory consolidation commands — no source code reading required.

---

#### AGENT.md — never loaded by the orchestrator; passed to subagents

**When it's used:** Only when the dispatched Claude session delegates work to a subagent via the Agent tool. The orchestrator reads SKILL.md to know *that* delegation is appropriate; it then passes the AGENT.md path to the subagent as its briefing.

**Why it's never loaded by the orchestrator:** AGENT.md contains step-by-step execution protocols, anti-patterns lists, CLI examples with full output, and detailed gotcha documentation. For `arc-failure-triage/AGENT.md` that's ~130 lines of investigation protocol (read failed tasks → check own code first → classify root cause). If this were always in the orchestrator's context, every task would pay for that token cost even when failure triage isn't happening.

**The split in practice:** `arc-failure-triage/SKILL.md` tells the orchestrator "this skill handles recurring failure patterns; load when a task subject contains 'Investigate recurring failure:' or 'Daily failure retrospective:'." The AGENT.md then tells the subagent step-by-step: run the scan, read error details, check our own code before blaming external services, write learnings to patterns.md.

**38% coverage is correct:** Only 40 of 105 skills have AGENT.md. Most sensors and infrastructure skills (fleet-*, arc-housekeeping, dispatch-watchdog) don't need subagent delegation — their tasks are simple enough for the orchestrator to handle directly with SKILL.md guidance.

---

#### cli.ts — the executable capability boundary

**How it's exposed:** Every `cli.ts` file is invoked via `arc skills run --name <skill> -- <args>`. The `arc` CLI wrapper runs `bun skills/<name>/cli.ts` and passes the remaining arguments. This is the "CLI-first" principle: if a capability doesn't have a CLI command, Arc can't use it in a task.

**How it's invoked in tasks:** The dispatched Claude session calls `arc skills run --name <skill> -- <command> --flag value` as a bash command. The cli.ts parses `process.argv.slice(2)`, executes the action, and prints output. The orchestrator reads stdout to determine what happened.

**Example — arc-email-sync:** `arc skills run --name email -- send --to whoabuddy@gmail.com --subject "Report" --body "..."` sends an email. The orchestrator doesn't know the Worker API URL or auth header — cli.ts handles all of that. The orchestrator only knows the CLI syntax from SKILL.md.

**Example — arc-housekeeping:** `arc skills run --name housekeeping -- check` outputs a JSON report. `arc skills run --name housekeeping -- fix` auto-repairs safe issues and commits. A P8 Haiku task running housekeeping doesn't need SKILL.md context for a simple `fix` run — the command is self-contained.

### The architectural reason for the split

SKILL.md is always-on cost. Every task with this skill in its `skills[]` array pays for SKILL.md tokens. If SKILL.md were 10k tokens, loading 5 skills would burn 50k tokens on overhead before the task even starts — exceeding the entire context budget.

AGENT.md is optional, on-demand cost. It's only passed to subagents that need it. A P8 housekeeping task never spawns a subagent. A P2 DeFi operation might spawn two (one for chain validation, one for position analysis), each receiving the relevant AGENT.md.

The practical result: the same framework handles "mark email as read" (3k context, 60 seconds, $0.01) and "create new skill from scratch" (25k context, 15 minutes, $2.00) without structural changes. The task's `skills[]` array and `priority` field determine what gets loaded and at what model tier.

### How Contact Loading Works Now vs. Before

**Before (pre-contacts skill):** Relationship context was implicit and unstructured. Agent identities lived only in `memory/MEMORY.md` (the fleet roster table). There was no cross-referencing between tasks and interaction history, no relationship graph, and no way to surface relevant contacts to a dispatch cycle automatically.

**Now (contacts skill + context command):** A dedicated dispatch integration command surfaces relevant contacts per task:

```bash
arc skills run --name contacts -- context --task-subject "<subject>" [--limit 10]
```

The `context` command tokenizes the task subject into keywords (3+ chars), scores every contact by how many tokens match across name fields, beat, notes, and handles, and returns compact markdown contact cards ranked by relevance. This output is injected into dispatch context when the `contacts` skill is loaded for a task.

**The gap:** The `contacts` skill is not automatically loaded for every task — it only appears when explicitly listed in a task's `skills` array. A task touching a known collaborator but without `contacts` in its skills list gets no relationship context at all.

### How Well Do Keywords Surface Relevant Contacts?

The `context` command's keyword scoring is lightweight but effective for its purpose. Assessment by scenario:

**Works well:**
- Task subject contains a name: `"Topaz Centaur quarterly review"` → scores Spark contact highly
- Task subject contains a handle: `"Reply to @arc0btc mention"` → returns Arc's own contact card
- Beat-based tasks: `"AIBTC Dev Tools signal"` → surfaces agents with `aibtc_beat = "Dev Tools"`

**Works poorly:**
- Indirect references: `"PR #109 review"` doesn't match any name field — contact 79 (the actual collaborator) won't score unless their notes field mentions "PR #109"
- Wallet-initiated tasks: `"Incoming x402 from SP2ABC..."` matches by STX address if the contact exists, but only exact address matching is supported; partial matching would fail
- Relationship-aware routing: "Who should review this ERC-8004 PR?" can't be answered from keywords alone — it requires querying `contact_links` for agents with `collaborator` relationships on ERC-8004 tasks

**Scoring blind spot:** The keyword scorer treats all fields equally — a mention of "arc" in a contact's `notes` field scores the same as "arc" in `display_name`. A field-weighted scoring system (name fields > beat > notes) would reduce noise from incidental mentions in notes.

### Gaps in Relationship Memory

**1. Interaction log is sparse.** The `contact_interactions` table exists but requires explicit `arc skills run --name contacts -- log` calls to populate. The reputation-tracker sensor creates BIP-322 reviews for PR interactions, but doesn't write to `contact_interactions`. An agent that has been reviewed 46 times may have zero entries in the interaction log — the two systems are siloed.

**2. No automated interaction logging from tasks.** When dispatch completes a task sourced from a known contact (e.g., an x402 message), no interaction record is automatically created. The interaction log has to be maintained manually or by explicit sensor logic. Result: the history in `contact_interactions` likely undercounts actual interactions significantly.

**3. Relationship freshness is invisible.** `contact_links` has no `last_active` or `last_interaction_at` field. A `collaborator` link created in January looks the same as one created yesterday. Over time, stale links dilute the signal — Arc can't answer "who have I worked with recently?"

**4. agent-hub and contacts are unlinked.** Hub capability data (skills, sensors, task volume, cost trends) doesn't flow into contact records for fleet agents. Knowing that Forge has 60 skills, 12 sensors, and handled 350 tasks last month is relevant context when deciding whether to route a task to them — but that data lives in `hub_agents`, not in the Forge contact record.

**5. No relationship traversal.** There's no way to query "all contacts two hops from Arc" or "all agents whose operator is a contact of whoabuddy." The `contact_links` bidirectional design supports traversal, but no CLI command exposes it. This limits graph-level reasoning about the network.

**6. External contact discovery is AIBTC-only.** The contacts sensor discovers agents via `https://aibtc.com/api/agents`. Contacts introduced through other vectors — GitHub PRs, email, x402 messages, direct fleet introductions — are not auto-populated. They exist as implicit knowledge (in memory notes, task summaries) but not as structured contacts.

### What Would Make Contact Tracking More Useful

**High value, low effort:**

1. **Wire reputation reviews → interaction log.** When `reputation-tracker` submits a BIP-322 review, also call `arc skills run --name contacts -- log` to write a `collaboration` interaction entry. One-line addition to `reputation-tracker/sensor.ts`. This would retroactively justify backfilling (the `log` command could accept `--occurred-at` for historical entries).

2. **Auto-log x402 interactions.** The `arc-payments` sensor already detects incoming STX/sBTC transfers. Adding a `contacts` lookup + interaction log write would capture economic interactions without manual effort.

3. **Add `last_interaction_at` to contact_links.** A computed column or periodic update showing when two contacts last had an interaction. Makes staleness visible and enables "recently active collaborators" queries.

**Medium value, medium effort:**

4. **Load `contacts` by default for tasks with known contact sources.** If a task's `source` field matches a contact's email, x_handle, or x402 endpoint, auto-add `contacts` to the task's `skills` array at creation time. Currently a sensor or human must explicitly include it.

5. **Field-weighted keyword scoring in `context`.** Weight name fields (display_name, aibtc_name, bns_name) 3× over notes and beat. Reduces noise from incidental mentions; increases precision for name-based lookups.

6. **Bridge agent-hub → contacts for fleet agents.** When the hub sensor runs, upsert `skill_count`, `sensor_count`, and `last_heartbeat` into the corresponding contact record for each fleet agent. Single join on `stx_address`.

**Lower priority:**

7. **Relationship traversal CLI.** Add `arc skills run --name contacts -- related --id <N> [--depth 2]` to walk `contact_links` to specified depth. Useful for onboarding new tasks that touch the ecosystem periphery.

## Section 5: arc0btc.com Monitoring and Uptime

*Completed task #5934, 2026-03-16T14:53Z*

*Sources: `skills/arc0btc-site-health/sensor.ts`, `skills/arc0btc-site-health/cli.ts`, `skills/arc0btc-site-health/SKILL.md`, `skills/worker-deploy/sensor.ts`, `skills/worker-deploy/SKILL.md`, `skills/site-consistency/sensor.ts`, `skills/site-consistency/SKILL.md`*

---

### 1. How is arc0btc.com currently monitored?

Three sensors touch arc0btc.com in some form:

| Sensor | Cadence | What it checks on arc0btc.com |
|--------|---------|-------------------------------|
| `arc0btc-site-health` | 30 min | **Nothing.** See naming issue below. |
| `worker-deploy` | 5 min | Detects local repo changes. Runs one `/health` check after deploy (non-fatal). Not periodic. |
| `site-consistency` | 1440 min (daily) | Reachability (HTTP 200), body contains "service", links to arc0.me, `.well-known/x402` exists |

**Summary: arc0btc.com gets one daily structural check and a post-deploy health ping. Nothing more.**

---

### 2. The arc0btc-site-health naming problem

`arc0btc-site-health` is the most prominent site-health skill in Arc's catalog. Its name suggests it monitors arc0btc.com. It does not.

The sensor checks:
1. `https://arc0.me` — HTTP 200 (uptime of the blog)
2. `https://arc0.me/api/posts.json` — JSON array (blog API health)
3. Latest post age ≤ 2 days (content freshness)
4. arc0me-site HEAD SHA vs. `worker-deploy` hook state (deploy drift)

All four checks target **arc0.me** (the blog site, hosted on Cloudflare Pages). The SKILL.md description is `"Monitors the arc0btc.com (arc0.me) site"` — the parenthetical reveals the confusion. This skill was written when the two domains were conceptually the same thing. They're now distinct:

- `arc0.me` — blog/signed-posts, Cloudflare Pages, Astro static site
- `arc0btc.com` — business/services, Cloudflare Worker, React SPA + Hono API + x402 payments

The 30-minute `arc0btc-site-health` sensor is healthy and correctly monitoring arc0.me. But **there is no equivalent monitoring for arc0btc.com**.

---

### 3. What the site-consistency sensor actually checks

The `site-consistency` sensor (daily) provides the only periodic check of arc0btc.com:

1. `https://arc0btc.com` returns HTTP 200
2. Response body contains `"service"` (case-insensitive substring match)
3. Response body contains `"arc0.me"` (cross-link check)
4. `https://arc0btc.com/.well-known/x402` returns HTTP 200

These are structural/liveness checks. They tell us the Worker is running and serving something. They do not tell us:
- Whether any paid endpoints accept requests correctly
- Whether the x402 payment flow produces valid payment requirements
- Whether `/health` returns valid JSON
- Whether response times are within acceptable ranges
- Whether the SSL certificate is healthy

---

### 4. How do we know the paid services actually work end-to-end?

**We don't — there is no end-to-end paid service verification.**

arc0btc.com runs three paid/agent-facing service paths:

| Endpoint | Purpose | Monitored? |
|----------|---------|-----------|
| `POST /api/ask-arc` | x402-gated LLM service (250 sats) | ❌ No periodic check |
| `GET/POST /api/research/*` | x402-gated research endpoints | ❌ No periodic check |
| `GET /agent.json` | Agent identity manifest | ❌ No periodic check |
| `GET /health` | Worker health state | ❌ Only checked once, post-deploy |

The `worker-deploy` CLI runs `fetch("https://arc0btc.com/health")` after a successful wrangler deploy. This check is:
1. **Non-fatal** — the code logs a warning on failure but doesn't fail the deploy task
2. **One-shot** — it's not repeated on any schedule
3. **Surface-level** — it only checks that `/health` returns 200, not that the response is valid JSON or that the reported state is healthy

For the x402 paid path specifically: the correct verification is a two-step probe. An agent should hit `/api/ask-arc` without payment, receive a 402 with `payment-required` header, verify the `PaymentRequiredV2` object contains a valid `payTo` address and `amount`, then optionally complete payment. This flow has never been verified automatically after the task #5933 migration to x402 v2 headers.

---

### 5. What test coverage exists?

**None in the CI/CD pipeline.** arc0btc-worker uses wrangler deploy; there is no GitHub Actions workflow running tests or health checks. The `worker-deploy/cli.ts` deploy pipeline is:
1. `npm run build:client` (Vite build — fails on TypeScript errors)
2. `npx wrangler deploy --env production`
3. Single non-fatal `/health` fetch

No unit tests, no integration tests, no smoke tests. The only feedback loop is: deploy succeeds or fails, and then humans or sensors notice if something is wrong.

---

### 6. Monitoring gaps and recommendations

**Gap 1 (High): No periodic health monitoring for arc0btc.com**

`arc0btc-site-health` should monitor arc0btc.com. Currently it monitors arc0.me. At minimum, add a `checkArc0btcHealth()` function to the sensor that hits:
- `https://arc0btc.com` (200 OK)
- `https://arc0btc.com/health` (200 + valid JSON)
- `https://arc0btc.com/agent.json` (200 + JSON with expected fields)

Or: rename the skill to `arc0me-site-health` and create a new `arc0btc-site-health` that actually monitors arc0btc.com.

**Gap 2 (High): No x402 payment probe**

After the x402 v2 migration (task #5933), there is no automated verification that the payment flow produces correct headers. A probe sensor should:
1. `GET /api/ask-arc` (expect 402)
2. Parse `payment-required` header → base64 decode → verify `PaymentRequiredV2` schema
3. Check `payTo` address matches Arc's known STX address
4. Check `amount` matches expected value (250 sats or equivalent)

This doesn't require spending funds — the probe only verifies the 402 response is correctly structured.

**Gap 3 (Medium): worker-deploy health check is non-fatal**

In `worker-deploy/cli.ts`, the post-deploy health check should fail the deploy task if `/health` returns a non-200. Currently it logs a warning and returns success. A failed health check after deploy is a strong signal the deploy broke something — it should surface as a task failure, not a warning log.

**Gap 4 (Medium): site-consistency runs once per day**

For a paid services site, daily is too infrequent. If the Cloudflare Worker crashes at 06:00 UTC and the daily sensor last ran at 05:00 UTC, the downtime goes undetected until 05:00 UTC the next day — a 23-hour window. This sensor should run every 30–60 minutes, matching arc0btc-site-health's cadence for arc0.me.

**Gap 5 (Low): No SSL certificate monitoring**

No sensor checks arc0btc.com's TLS certificate expiry. Cloudflare manages certs automatically, but if the domain renewal lapses or the Worker is misconfigured, no alert fires. A certificate check (using `https://arc0btc.com` with TLS error catching) would surface this before it becomes user-visible.

| Priority | Action | Effort | Impact |
|----------|--------|--------|--------|
| High | Add arc0btc.com health checks to `arc0btc-site-health` sensor (or create dedicated sensor) | 2h | Closes the main monitoring blind spot |

#### Principle 1 — Make Requirements Less Dumb

**Budget ceiling mismatch: DAILY_BUDGET_USD=500 vs. D4=$200/day**

**SENSOR_TIMEOUT_MS is uniform at 90s regardless of sensor complexity**

`sensors.ts:196`:

#### Principle 2 — Delete the Part/Step/Process

**`shouldRun()` is a subset of `claimSensorRun()` — one of them is redundant**

`sensors.ts` exports both:
```typescript
export async function shouldRun(name, intervalMinutes): Promise<boolean>    // check-only, no claim
export async function claimSensorRun(name, intervalMinutes): Promise<boolean>  // check + write
```

`shouldRun` reads state and returns a boolean without updating `last_ran`. It's useful if a sensor wants to check before doing expensive work and then separately claims. In practice, all three sensor-runner paths use `claimSensorRun` or `createTaskIfDue` — they claim atomically. Individual sensor files that call `shouldRun` without claiming create a race condition (two sensors could both return `true` before either claims). Audit: `shouldRun` should be removed if no sensor uses it for multi-step check-then-claim patterns. If it is used that way, the race condition should be documented.

**`git()` and `runCommand()` in safe-commit.ts are the same function**

`safe-commit.ts`:
```typescript
export async function git(...args: string[]) {
  const proc = Bun.spawn(["git", ...args], { cwd: ROOT, stdout: "pipe", stderr: "pipe" });

}
async function runCommand(cmd: string, args: string[]) {
  const proc = Bun.spawn([cmd, ...args], { cwd: ROOT, stdout: "pipe", stderr: "pipe" });
  ...
}
```

Identical bodies. `git` is `runCommand` with `"git"` as the first arg. Delete `runCommand`; replace call sites with `git` or a renamed `spawnCapture(cmd, ...args)` if non-git commands are needed.

**`db` and `dbWrite` in web.ts are two names for one singleton**


`getDatabase()` returns a module-level singleton — both variables point to the same `Database` instance. SQLite doesn't require separate connections for reads vs. writes. The two names imply isolation that doesn't exist. Delete `dbWrite`, use `db` throughout. The naming misdirects anyone reading the file about the concurrency model.

**`computeSkillHashes()` re-reads SKILL.md files already read by `resolveSkillContext()`**

1. `resolveSkillContext(skillNames)` — reads `skills/<name>/SKILL.md` for each skill
2. `computeSkillHashes(skillNames)` — reads the same files again to compute SHA-256 hashes

Two sequential passes over the same files. `readFile()` uses synchronous `readFileSync` — the second read may hit the OS page cache, but it's still redundant work and adds latency. A single pass that returns both content and hash would eliminate the double read.

#### Principle 3 — Simplify and Optimize

**N+1 query in `buildParentChain()`**

`dispatch.ts:239–250`:
```typescript
function buildParentChain(task: Task): string {
  const chain: string[] = [];
  let current = task.parent_id ? getTaskById(task.parent_id) : null;
  while (current !== null) {
    chain.push(`  #${current.id}: ${current.subject} (${current.status})`);
    current = current.parent_id ? getTaskById(current.parent_id) : null;
    if (chain.length >= 10) break;
  }
```

Each iteration is a separate `getTaskById()` call — up to 10 individual DB reads. For a 3-level parent chain (the most common case for audit tasks like #5936 → #5910), that's 3 round trips where one would do. A single recursive CTE resolves the entire chain in one query:

```sql
WITH RECURSIVE chain AS (
  SELECT id, subject, status, parent_id, 1 as depth
  FROM tasks WHERE id = ?
  UNION ALL
  SELECT t.id, t.subject, t.status, t.parent_id, c.depth + 1
  FROM tasks t JOIN chain c ON t.id = c.parent_id
  WHERE c.depth < 10
)
SELECT id, subject, status FROM chain WHERE id != ? ORDER BY depth DESC
```

Low-traffic code path but illustrates a pattern to avoid in future DB helpers.

**`processLine()` dual accumulation paths in dispatch stream parser**

`dispatch.ts` `processLine` function handles two result-accumulating paths without `else if`:

```typescript
}
if (parsed["type"] === "assistant") {
  // appends full content blocks to result
}
```

These event types are mutually exclusive in Claude's stream-JSON format (a line can't be both). But the two `if` statements instead of `if/else if` leaves an implicit assumption uncommented. If Claude's format ever emits a combined type, text would be double-accumulated. Minimal risk, but `else if` is both clearer and marginally faster. The `"result"` type handler already uses separate `if` correctly. Worth the one-character fix.

**arc0btc.com — React SPA via Cloudflare Worker**

The React SPA is in a separate repo (`arc0btc-worker`) not directly accessible here. Observations from code references in earlier audit sections and prior tasks:

**Likely large bundle: Mermaid.js in Architecture.tsx**

Task #5932 added `Architecture.tsx` which renders a Mermaid state machine diagram. Mermaid.js is ~800KB minified. Cloudflare Workers have a 1MB compressed bundle limit. If Mermaid is imported at the top of `Architecture.tsx` and bundled eagerly:
- The entire SPA pays ~800KB for a diagram that appears on one route
- Workers' 1MB limit may be approached

**Fix:** Dynamic import with `React.lazy` + `Suspense`:
```tsx
// Instead of: import mermaid from 'mermaid'
const MermaidDiagram = React.lazy(() => import('./MermaidDiagram'));
```

Or: use a pre-rendered SVG in the component (the diagram is static — it doesn't change at runtime). Static SVG eliminates the Mermaid runtime entirely.

**AskArcForm — stateful, correct use of React**

The x402 payment form component manages: loading state, payment-required header parsing, payment signature submission, and result display. This is genuinely stateful UI — React is the right tool. No performance concerns identified from the interface description.

**Component count is small (Architecture, Legal, Services, AskArcForm, Layout, App)**

With ~6 components and a SPA that serves one operator plus occasional external agents, React render performance is not a concern. No unnecessary re-renders identified from component descriptions. The main optimization opportunity is the Mermaid bundle, not component rendering.
