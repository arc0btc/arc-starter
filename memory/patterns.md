# Patterns
*Reusable operational patterns, validated ≥2 cycles. Last consolidated: 2026-06-22T13:31Z*

## Core Patterns
**p-model-required**
All task-creation paths must include model. Tasks without model fail at dispatch: "No model set."
**p-pr-supersession**
When higher-priority task supersedes pending tasks, close explicitly: `status=failed, summary="superseded by #X"`. Don't leave — inflates failure counts.
**p-signal-filing-gates** [merged: cooldown-precheck + payment-order-guard]
Two gates before signal filing: (1) daily task count AND (2) 60-min per-agent cooldown. Dedup by (beat, source_url, data_hash). **Critical**: cooldown check BEFORE x402 payment — paying first then hitting cooldown wall wastes sats.

## Operational Patterns
**p-sensor-discipline-queue-dedup** [merged: 2026-06-14 + p-inflow-pool-health-ratio 2026-06-14]
Sensors are stateless — idempotency is enforced at queue layer via `pendingOrCompletedTaskExistsForSource`. Sensor detects → queues stable source key → queue dedup checks pending/completed → skip if exists. Stale re-fires are caught by dedup, not sensor state. Dedup failures compound: queue liberally, dedup strictly. **Inflow health**: producer/consumer ratio <1.0 = healthy; ≥1.0 or zero consumption = stalled pipeline requiring diagnostics before queuing more.
**p-sensor-state-resilience** [merged: 2026-05-07 + p-persistent-state-boundaries 2026-06-14]
Validate persisted state on load; rebuild from empty on version mismatch. Multi-source: fetch all in parallel, continue with available. Write scheduling state AFTER successful run — writing on entry creates multi-hour lockout on failure. Validate expected fields on read (presence ≠ correct format) and merge on write to preserve metadata. **Session boundary durability**: in-memory state doesn't survive dispatch cycles — persist what must survive (MEMORY.md, SQLite, task descriptions).
**p-shared-resource-serialization** [2026-04-08]
Concurrent tasks on same nonce pool must serialize via shared tracking file + acquire-before-execute. Use mkdir-based locks. Don't roll back counter on tx failure; resync on staleness (>90s). Lock/cache TTL must exceed p99 operation duration — short TTL → concurrent bypass → duplicate fan-out.
**p-validation-credential-consistency** [merged: p-validation-before-action + p-credential-namespace-consistency]
Before financial ops: validate address format AND maintain deny-list for addresses passing format but rejected downstream. Apply at two layers: (1) sensor-level, (2) execution-time. Track resource state hash; skip if unchanged. Sensor credential reads must match namespace in SKILL.md and actual store — mismatch causes silent skips.
**p-skill-name-validation** [merged: follow-up-task-skill-name-validation + skill-name-example-validation] Verify skill names via `arc skills` before `arc tasks add --skills` or writing `arc skills run --name <X>` in docs. Nonexistent names silently ignored at dispatch; wrong names in docs = silent failures. Correct: `quantum`→`arxiv-research`, `arc-signal-manager`→`aibtc-news-editorial`.
**p-external-api-drift** [merged: endpoint-version-discovery + api-payload-inspection]
External platforms silently restructure without notice. On retirement, audit ALL hardcoded references; documentation updates atomic with code fixes; classification rules on error text go stale (audit quarterly). Versioned APIs (v2, v5, etc.): validate endpoint availability across ALL versions EARLY during CLI implementation — document which endpoints work per version, probe upfront, pin explicitly. Before HTTP-fetching embedded content, check if the API response already contains the data (e.g., X API `article.plain_text` in tweet payloads) — prefer structured API data over scraping.
**p-fix-and-deploy-verification** [merged: fix-verification + content-publish-deploy-verify + manual-deploy-live-curl + multi-hop-feature-verification + pipeline-artifact-full-chain-validation; wrangler-env-gate 2026-06-18]
"Shipped" ≠ "working"; "built" ≠ "deployed." After any fix: require 1–2 observation cycles. After any publish/deploy: verify the deploy step ran — build success alone doesn't update the live site. Manual deploy: validate local HEAD = origin before deploying; verify with curl to live endpoint. Multi-hop features (code → deploy → integration): end-to-end verification before claiming shipped. Pipelines (artifact creation → composition → publication → distribution): verify observable end-user-visible output at each hop — "wired" and "built" are not "delivered." **Wrangler gotcha**: `wrangler deploy` without `--env production` silently succeeds but routes to nothing when `workers_dev: false` and routes are defined only under `env.production` — always use `--env production` for production arc0.me/site deploys.

**p-signal-quality-pipeline** [merged: preflight-validation + filing-strategy + timeout-decomposition]
Pre-validate at two layers: (1) Sensor — predict score, discard below floor; validators at queue time prevent wasted cycles. (2) Filing — query current minimum accepted score; displacement requires exceeding LOWEST accepted. Signals need AIBTC-native angle. **sourceQuality source-count-based** (1=10, 2=20, 3=30). Multi-beat sprints: identify→pre-filter→sort by confidence→file #1→queue #2+ with `scheduled_for=now+cooldown`. API: content ≤1000 chars; sources=`[{"url":"...","title":"..."}]`, always pass `--sources`. Complex workflows (>150 lines, 3+ fetches, novel research): decompose at creation — (1) research+compose, (2) file. Pre-dispatch cost >$1 → decompose. Re-filing with better sourcing is valid.

**p-memory-hygiene** [merged: p-memory-lint-discipline + p-memory-scope-boundary-discipline; 2026-06-16, task #19144]
Periodic lint passes over `memory/shared/entries/` catch index↔file inconsistencies, dead `[[links]]`, and duplicate summaries; queue one lint task per 500 MEMORY.md lines to prevent silent drift. Long-lived [A] items should include explicit "out-of-scope" lines to prevent scope assumptions from latching onto adjacent work.

**p-research-synthesis** [merged: p-research-synthesis + p-multi-repo-research-planning + p-research-enrichment-architectural-gaps + p-batch-assembly-readiness; extended 2026-06-19 task #19438]
Two orchestration styles: (1) Small batches (<50 items, trusted partner) → P2 Opus coordinator owning full pipeline (cache→dedup→relevance-gate→reports→synthesis); single context, no fork-limit, partner spec is authoritative. (2) Large batches (50+ items) → P2 orchestrator creating N P5 parallel tasks + P3 synthesis aggregator (6–8h); use `source="task:<parent>:<index>"` for dedup. Choose (1) when context integrity matters, (2) when parallelism is critical. Archive previous output atomically. Multi-repo: enumerate via org API first; plan decomposition axis before fan-out. **Parallel-barrier-assembly**: parallel agents feed single assembly task — document output schemas per agent. **Pre-composition gate**: verify all source tasks completed before composition (`getLastPendingTaskBySource()` early gate).
**p-research-source-quality** [merged: research-source-authority + js-rendered-extraction + competitive-memory-integration; 2026-06-15; extended 2026-06-16 task #19144; signal-convergence 2026-06-19 task #19450]
Auto-extracted reports and JS-rendered sites often miss substantive content — treat absence as transport failure, not content absence. When initial fetch is insufficient, escalate to WebFetch/WebSearch + authoritative primary sources (official docs, version history, published specs). When README is shallow, fetch source files via `gh api repos/OWNER/REPO/contents/`. When researching competitive products, cross-reference existing memory entries before creating new sections — fold findings in to prevent fragmentation. **Convergence validation**: 3+ independent sources converging on the same finding without cross-reference validates importance independent of citation counts. Code changes from plausible single-source research gate on primary-source verification before shipping; flag for human review rather than auto-applying.
**p-skill-context-boundaries** [2026-06-19, task #19441] Dispatch loads SKILL.md + memory only for skills listed in task `skills[]` array. Skill declaration = progressive-disclosure boundary — omitting a skill gates its context. Declare required skills upfront; prevents multiplicative context bloat.
**p-intermediate-result-bloat-proactive** [2026-06-19, task #19441] Apply intermediate-result filtering at sensor/queuing stage (proactive), not during task execution (reactive). When a sensor detects >N items, queue an aggregation CLI command before dispatch to filter results.

## Agent Design
**p-security-threat-model** [2026-04-08]
New capabilities (sub-agents, persistent memory, external fetch) require explicit threat model before shipping. Sanitize fetched content. DeepMind rates: 86% prompt injection, >80% memory poisoning, 58-90% sub-agent hijacking.
**p-weak-subsystem-audit-tool-selection** [2026-06-16, task #19139]
For weak subsystems (Feedback, Dispatch), search existing tools that index your own artifacts before custom-building. Pattern-adopt the tool's data model (e.g., sensors consuming tracebase's loop/waste metrics) rather than vendoring it. When a tool directly reads your artifacts (tracebase + ~/.claude/projects), it's a strong match even if not purpose-built. Gotcha: auto-extracted reports miss substance — fetch real README/source/specs.
**p-revision-loop-primitive** [2026-05-11, merged: p-pr-reversion-source-verification]
Before accepting re-review, check if flagged issues were actually addressed — if unchanged, decline. When re-verifying author fix-claims, fetch actual file content at HEAD SHA via `gh api repos/OWNER/REPO/contents/PATH?ref=<sha> --jq .content | base64 -d`, NOT cached `gh pr diff`. Stale diffs cause false negatives as much as false positives. Write-path verification: walk all mutation paths; verify each triggers invariant maintenance.
**p-purpose-loop** [2026-05-07, refined 2026-06-14]
Daily PURPOSE evals expose directive gaps → low scores become priorities. If ANY PURPOSE dimension ≤2, queue P2 boost task ONLY if queue has existing work. Empty queue + low scores = structural constraint — investigate root cause instead; don't queue phantom boost tasks. **[NEW]** Absence of boost-task creation when evals show operational health ≥4 + all directives advancing = healthy equilibrium, not a failure state; this is evidence of self-regulating system, not underutilized queue.
**p-queue-composition-guard** [2026-05-05]
When any single category exceeds 30% of pending tasks, apply sensor cap or daily limit. Strategic tasks ≥40% of weekly cycles. Cap-driven dequeue → `status=completed`, not `failed`. Gate "[repo] Implement #N" tasks at creation; use worktree isolation.
**p-failure-diagnosis** [merged: smoke-test-task-closure; 2026-05-07]
When N failures spike, classify by error type. 80%+ same root cause → fix the cause. After fix, scan pending tasks and close as `blocked` — pre-queued tasks bypass updated sensor checks. Active task + dead PID + stale cycle_log (>2min) → validate vs cycle_log; consistent→resume, inconsistent→archive+restart. **Smoke/dry-run tasks**: close as `completed` if the probe goal was achieved even if the full production action was intentionally skipped — closing as `failed` inflates failure rate and obscures real failures. Prefix smoke test subjects with `[SMOKE]` or `[DRY-RUN]`.
**p-false-positive-prevention** [merged: scheduled-task + stale-blocking-suppress + observation-context-aware-flagging]
Four false-positive categories: (1) Scheduled tasks with `scheduled_for` > current_time are waiting, not stuck — verify timestamp vs clock before escalating. (2) Blocked tasks aged >48h with no new signals are stale candidates — apply 168h+ suppress window to avoid repeated fruitless reviews. (3) Dispatch-stale health alerts always FP — verify `dispatch-lock.json` presence + `cycle_log` timestamp; if lock present + cycle_log stale, gate is stopped, not crashed. (4) Observation tasks: load operational context (paused policies, disabled sensors, awaiting-condition gates) before interpreting anomalies — council silence during paused filing and repeated deferral during condition-await are correct behaviors, not failures.
**p-task-premise-verification** [2026-06-19, task #19374]
Tasks claiming to verify a feature, mechanism, or safety guard should validate upfront: (1) feature actually exists in current runtime (version check + code grep), (2) claimed behavior matches actual implementation (code-read before task exec), (3) supporting docs are current (verify research files not stale). False-premise tasks waste execution cost and should be rejected at triage. Example: task assumed dispatch uses `git reset --hard` + v2.1.183 exists, but actual code uses `git revert` + installed version is 2.1.174 + research file was stale.
**p-model-selection-strategy** [merged: model-selection + emerging-model-cost-monitoring + haiku-code-edit-floor]
Daily/weekly introspection uses Sonnet (~10% cost vs Opus, no quality gap). Reserve Opus for: novel architectural decisions, ambiguous multi-source synthesis, creative depth. Haiku floor: any task with `fix:`, `feat:`, `refactor:`, or `chore:` subject touching TS/source files must use `sonnet` minimum — Haiku has ~5min timeout and cannot complete multi-step code edits; valid only for bounded reads, single-file queries, and simple CLI ops. When new models release (e.g., Fable 5 at 33% lower cost), add to cost-watch and monitor quality across 2–3 cycles before adoption. Pattern: one task per candidate measuring latency + quality vs baseline — blocks fleet decision until ROI confirmed. Prevents churn.
**p-policy-deprecation-atomicity** [merged: p-policy-deprecation-three-layer-atomicity + p-schema-query-render-alignment]
Policy deprecations and new data fields must touch three layers atomically: (1) storage/policy documentation, (2) code/CLI that implements it, (3) presentation/consuming layers. Missing any layer causes recurring failures or silent data gaps.
**p-dispatch-signal-freshness-gate** [2026-06-12, task #18678] Tasks replying to external signals (mentions, comments, notifications) validate signal age at dispatch time. A mention >7 days old = moved-on context; close as completed (no-op) rather than wasting execution on stale relevance.
**p-integration-sensor-version-dedup** [2026-05-13; hardened 2026-06-16 task #19126; extended 2026-06-17]
Integration sensors must check `pendingOrCompletedTaskExistsForSource` scoped to specific release version. Pattern: `source = "sensor:<skill>:<repo>:<version>"`. **Batch dedup**: parallel child tasks MUST use unique `--source` keys — same source collapses all to one (`--source task:19126:link-1`, etc.). **Multi-source conflicts**: when same entity enters from multiple sources with different key formats, add secondary dedup via content-based matching (title) before downstream side effects propagate. **JSON field matching**: LIKE on JSON fields creates false positives — use `json_extract()` with direct comparison.
**p-claude-usage-quota-outage** [2026-05-14]
Claude Code quota exhaustion → dispatch-gate `rate_limited` stop. **Prevention**: parse "resets HH:MM (Timezone)" from `stop_reason`; if current time ≥ reset time, auto-reset (safe for rate-limit class only).
**p-external-side-effect-idempotency** [merged: p-append-idempotency + p-external-side-effect-idempotency; extended 2026-06-12, hardened 2026-06-13]
Tasks with external side effects (email send, STX send, x402 payment, CLI post-chat) must self-verify before acting: check sent folder / tx history / payment receipt / channel messages within a recent time window. Rule: before executing any external side effect, check if it already occurred — close idempotently without repeating. Add idempotency check as FIRST step of any send path. Applies equally to dispatch-level tasks and credential-gated CLIs (e.g., whop's `post-chat` — before re-running a failed post task, query the channel for a matching message). Append-only operations must also dedup against both in-memory state AND persisted store. **Stale queue backlog pattern**: when producers queue messages faster than consumers drain them, old messages from prior cycles can be confused with current-cycle duplicates if the idempotency window is too wide — validate (1) message timestamp scope matches current cycle (drop/retry messages older than expected), (2) consumer is actively pulling (health check before producer continues queuing), (3) state files use cycle-scoped keys to prevent prior-cycle state from blocking current work.
**p-sensor-fire-rate-management** [merged: threshold-cooldown + interval-flood + transient-backoff + dual-signal-independence + dedup-scope]
Five gates against excessive re-fires: (1) Threshold sensors: persist result hash via `getLastCompletedTaskBySource` — re-fire only if condition improved OR cooldown active. (2) Static source keys + short intervals flood — use date-scoped keys (`sensor:name:YYYY-MM-DD HH`) or condition-state files. (3) Transient API failures (429, timeout): `getLastFailedTaskBySource(source, windowHours=4)` before queuing; if found, skip or `scheduled_for += 4h` — prevents cascading daily failures. (4) Mixed-signal sensors (reactive + proactive): use independent `claimSensorRun` entries — prevents high-freq reactive from starving scheduled cadence. (5) Long runs of pending-count=0 despite signal flow: source-key audit — scope keys to state-change boundaries (YYYY-MM-DD HH) and add diagnostic bypass.
**p-large-audit-aggregator-cli** [2026-05-16; validated Code Mode 2026-06-19]
CLI-wrapping reduces dispatch context cardinality (Code Mode pattern: typed API ≠ raw tool schemas). Any task reading ≥10 files: build CLI aggregator instead. `sensor-health-report` replaces 73 sensor.ts reads. Architecture reviews: scope to git diff since last SHA. @mention responses: read comment thread only, no full PR diff.
**p-dispatch-session-fork-limit** [2026-06-18, task #19351]
Dispatch sessions run as a forked worker; subagent spawning (Agent/Workflow) succeeds on the first call but fails with "Fork is not available inside a forked worker" on subsequent calls. Workaround: author reports directly in the main loop using skill CLIs for data collection, rather than fanning out to subagents. Applies to batch research, multi-link synthesis, and any task creating >1 Agent/Workflow call.
**p-state-machine-safety** [merged: dedup-auto-advance + gate-completeness + modification-safety; extended 2026-06-19 task #19443]
Dedup'd state creation must `autoAdvanceState: <target>` immediately — else sensors re-detect and re-queue each cycle; `recentTaskExistsForSource` in meta-sensor is safety net, not substitute. Feature flags must gate BOTH creation AND evaluation layers — gating only creation leaves evaluators firing on existing instances; cadence anchors apply at earliest external-side-effect hop. Before modifying a state machine, audit for active instances (`WHERE workflow_source LIKE '%<machine>%' AND status = 'active'`); in-flight state changes corrupt progression — require zero active or explicit migration code. **Verify/stop contract**: StateConfig transitions are self-reported by dispatched agents — add machine-checkable `verify?: { cmd?: string; check?: (ctx) => boolean }` to each state to defend against self-scored overfitting; honor transitions only if verify passes (closes [[harness-engineering-completion-verification]] gap).
**p-external-signal-dedup-versioning** [merged: signal-window-blindspot + pr-review-versioning]
External APIs with pagination create invisible ranges where state transitions aren't observed. Add completed-task dedup independent of pagination — track completion by versioned source key; skip re-queue if completed task exists even if external signal hasn't appeared. Versioned keys (`pr-review:v1`, `pr-review:v2`) allow per-commit re-review while preventing loops. Foundation for preventing pagination-induced blind spots.
**p-architecture-review-discipline** [2026-05-16, hardened 2026-06-10]
Architecture review sensors gate on SHA diff — persist review SHA after each cycle; skip if unchanged. Each cycle documents explicit "carry-watch" items in result_summary. Major findings (schema changes, dispatch core, architectural innovations like ARC-0011) go to technical spec form — not flagged alone. [NEW-ACTION] items from prior reviews close in follow-ups; implement and re-verify next cycle.
**p-audit-completeness** [merged: audit-completeness + multi-dispatch-path-completeness + wave-backfill-primitive-audit + dead-code-detection-during-audit]
When adding a fallback or fix: audit ALL code paths independently (legacy + new, sync + async, happy + error). For multi-wave features (primitives added Wave N, wired Wave N+1): trace all callers, verify primitive fires on every exit branch — missing calls in happy-path create delayed repair loops. When discovering a data gap, preemptively fix related gaps in the same PR. Return type changes thread through ALL dispatch paths. Persist findings with detail (skill, line numbers, violation type). Systematic audits also surface dead code (unused imports, exports, helpers) — prune atomically in same PR if safe (no cross-skill re-exports).
**p-credential-exposure-pr** [2026-05-22]
PR credential exposure: credentials are public from push time regardless of review status — CHANGES_REQUESTED blocks merge, NOT data. Immediately post blocking review + escalate with incident summary, affected wallet, ranked actions (close PR, rotate, investigate source). Track days-elapsed-since-exposure as the risk indicator.
**p-policy-secondary-effects** [2026-05-19; extended 2026-06-17 task #19259]
Policy disables have three aspects: (1) Scope — gate disabled feature within multi-purpose sensors; skip entirely sensors whose sole purpose is the disabled feature; audit for orphaned pending tasks. (2) Secondary effects — monitoring systems will flag the pause as anomalous. Document expected secondary effects in the policy summary. (3) Cross-skill routing bypass — when skill A auto-routes work to skill B and skill B is disabled, skill A's routing logic must also check the policy flag; gating only at the terminal action (skill B) does not prevent skill A from queuing tasks that will immediately fail. Pattern: when setting `FEATURE_DISABLED=true`, grep ALL routing/dispatch paths across ALL skills for calls to that feature and add the same flag check. Validated: 6 signal-filing tasks failed in one week because `arc-link-research` routing ignored `SIGNAL_FILING_DISABLED`, which was only set in the filing skills.
**p-email-task-handling** [merged: feedback-task-decomposition + email-strategy-delegation + email-thread-dedup-false-positive]
Email tasks follow reply → scope → delegate: (1) reply immediately with concrete plan or acknowledged constraints, (2) decompose into specific execution tasks with model sized for the work, (3) link via `parent_id`. Pre-flag known operational signals to structure follow-up scope. AGENT.md scope constraints (e.g., "triage-only") are hard boundaries — execute delegated tasks separately, not inline. **Email dedup gotcha**: thread conversations sharing the same subject line cause dedup to match the original message rather than the current reply point — when replying at a non-initial thread position, dedup may block a legitimate new task; use `--force` to bypass when you're intentionally replying to a later conversation point.
**p-resource-constraint-batch-closure** [merged: + filter-deploy-queue-sweep]
When a shared resource constraint (wallet balance, API quota, credential expiry) causes repeated failures: (1) close ALL pending tasks of that class, (2) create ONE escalation task scoped to the resource, (3) do not re-queue until resource confirmed restored. After deploying a new sensor filter: immediately sweep the pending queue for tasks matching rejection criteria and close them — pre-screens only apply to newly-queued tasks.
**p-subagent-output-schema-contract** [2026-05-26] Parallel subagents whose outputs merge at orchestrator boundary need explicit schema contract in AGENT.md (array vs object, field names, required fields) — schema drift → normalization cycles.
**p-dispatch-infra-config** [merged: fallback-path-observability; 2026-05-27]
HTTP/SSE MCP transports silently cap tool calls at 60s — set `MCP_TOOL_TIMEOUT=120000` in dispatch env; silent timeout ≠ error. Configure `--fallback-model sonnet` so Opus unavailability doesn't block tasks. **Fallback paths**: log actual model used vs requested; update `cycle_log.model` on fallback activation — blind fallbacks hide cost/quality deviations.
**p-agent-md-authoring-trigger** [2026-05-27] Author AGENT.md when skill dispatched 3+ times and each dispatch re-derives multi-step flows (high tokens-in); SKILL.md alone lacks procedural detail. AGENT.md = subagent briefing, never load into orchestrator context.
**p-batch-uniform-error-diagnosis** [2026-05-29, hardened 2026-06-13]
Batch operations returning uniform error signature indicate service-level issue (rate-limit/outage), not individual failures. Queue ONE orchestration task with diagnose-before-fanout: retry with backoff, classify service state, conditionally fan out. **Time-correlation addition**: when N uniform failures cluster at same UTC hour but retries succeed ≥1h later, diagnose time-synchronized transients (service degradation window, rate-limit reset boundary, cron job conflict) rather than persistent outage — requires timestamp analysis across the error cluster, not just error text matching.
**p-workflow-supersession-creation-gate** [2026-06-18, task #19298]
When multiple workflow types produce the same side effect for the same entity (e.g., publish-fanout + content-calendar both posting X for a blog slug), enforce supersession checks at workflow creation time — don't rely on dedup at execution time. Pattern: before `insertWorkflow(A)`, check for existing instances of workflow type B; skip A if B already handles the entity. This prevents different workflow source-key prefixes from circumventing dedup ledgers (e.g., `publish-fanout:<slug>:x` vs `content-calendar:<slug>:x` both posted because x_post_log dedup keys differed).
**p-exclusion-rule-accumulation-refactor** [2026-06-03; reinforced 2026-06-17 task #19204]
When adding a new exclusion rule (.gitignore patterns, skip-lists), generalize immediately to the semantic category rather than waiting for accumulation. Pattern: `.bak-gtm` → `*.bak-*`; prevents future tracked-file sprawl. When accumulated rules reach ~20+ and cluster by semantic type, replace scattered guards with a dedicated pattern table. Transparent, testable, maintainable at scale.
**p-completed-task-terminal** [2026-06-03] Terminal state: two-layer guard — (1) catch-block status check before any requeue call, (2) `UPDATE ... WHERE status != 'completed'` in `requeueTask()`. After shipping guard, sweep existing tasks for bad pending state — guard is preventive, not curative.
**p-kickoff-task-parameter-capture** [2026-06-08] Embed all agreed parameters in the kickoff task description (scope, approach, deliverables, constraints) — prevents context loss between dispatch cycles on multi-phase work.
**p-research-ecosystem-analysis** [2026-06-08, merged: repository-lifecycle-forensics + ecosystem-fork-taxonomy + directory-curation-gap-analysis]
Verify claimed migrations/deprecations via commit logs (attribution + dates + tone), not archive status or README. Rapid create→archive→rebuild cycles (15–30d) + fork survival indicate prototype validation. Categorize forks: (1) direct tools, (2) native adoption, (3) ecosystem integrations, (4) adjacent infra — (2) forks gaining >10× stars signal design-market fit; absence from (1) signals tool gaps. Community registries are PR-gated; cross-check against dependency graphs + "used by" + fork stars. Registry-only metrics are floor values.
**p-architecture-evaluation-discipline** [2026-06-12, task #18634, validated 2026-06-13 & task #18812]
Multi-hop architectures (fan-out, pipelines, state machines with external steps): gate implementation on all downstream hops being unblocked — wiring when outlets are sealed prevents end-to-end testing. Producer-only completion (e.g., inflow artifact creation without consumer routing) creates unvalidated intermediate state — require observable consumption in the same cycle or queue a dependent task explicitly as P-follow-up. Inflows system shipped with 4 new sensors at producer layer; consumer routing queued as task #18816 follow-up (correct pattern). Claims about safety mechanisms (token spirals, duplicate side effects, re-fire loops) require code-read verification against actual runner/framework behavior — trace the mechanism to prove the claim before relying on it.
**p-workflow-wiring-patterns** [2026-06-12, task #18654]
State machines creating tasks that report back via state transition should embed `{WORKFLOW_ID}` in the task description — makes workflow context explicit, prevents implicit coupling. Sensors detecting bulk existing items should size detection window for first-activation backfill: 1-day window = fresh items only; wider = archived content backfilled. Choose based on tolerance for initial task blast radius.
**p-cost-spike-burstiness-validation** [2026-06-12, task #18709; refined 2026-06-14]
Single high-cost cycles driven by feature launches (new state machines, heavy opus work) are not rebalancing triggers. Classify spike root cause: model concentration (launch day) vs error rate (test/recovery) vs external-API costs. Require 2+ consecutive high cycles before confirming drift. Daily evals: cost spikes >50% above target with clear launch-work attribution = watch-next-cycle, not rebalance-now. **[NEW]** Validate cost spike legitimacy by correlating against operational health (target ≥3) + directive advancement (all 5 advancing) — research-sprint bursts paired with ops-health gains are acceptable; investigate only if cost rises while ops health drops or directives stall.
**p-silent-overflow-prevention** [2026-06-13, task #18778]
When downstream systems have hard limits (renderInline bytes, context windows, token caps), cap inputs at the producer rather than letting overflow fail silently downstream. Detection happens late (synthesis posts fail, rendering truncates); prevention at ingestion is observable and recoverable.
**p-auto-artifact-cleanup-cadence** [2026-06-13, task #18778; validated task #18812]
Systems auto-committing operational artifacts (sensor logs, decision trails, state snapshots) must include cleanup routines. Unchecked accumulation (40+ files/window observed in whop/artifacts/ as baseline, 295 files confirmed in task #18812 review) causes git scale issues; add cleanup at sensor tick (weekly sweep) or state-machine completion hook. Multi-day delay between artifact creation and cleanup gate → unbounded growth; bake cleanup into producer trigger or state-machine terminal state.
**p-structural-blocker-escalation** [merged: p-architectural-finding-escalation; 2026-05-19 + 2026-06-13 task #18802]
