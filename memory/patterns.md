# Patterns
*Reusable operational patterns, validated ≥2 cycles. Last consolidated: 2026-06-27T18:12Z*

## Core Patterns
**p-model-required**
All task-creation paths must include model. Tasks without model fail at dispatch: "No model set."
**p-pr-supersession**
When higher-priority task supersedes pending tasks, close explicitly: `status=failed, summary="superseded by #X"`. Don't leave — inflates failure counts.
**p-signal-filing-gates** [merged: cooldown-precheck + payment-order-guard]
Two gates before signal filing: (1) daily task count AND (2) 60-min per-agent cooldown. Dedup by (beat, source_url, data_hash). **Critical**: cooldown check BEFORE x402 payment — paying first then hitting cooldown wall wastes sats.

## Operational Patterns
**p-sensor-discipline-queue-dedup** [merged: 2026-06-14 + p-inflow-pool-health-ratio 2026-06-14]
Sensors are stateless — idempotency is enforced at queue layer via `pendingOrCompletedTaskExistsForSource`. Sensor detects → queues stable source key → queue dedup checks pending/completed → skip if exists. Stale re-fires are caught by dedup, not sensor state. Dedup failures compound: queue liberally, dedup strictly. **Inflow health**: producer/consumer ratio <1.0 = healthy; ≥1.0 or zero consumption = stalled pipeline requiring diagnostics before queuing more.
**p-sensor-state-resilience** [merged: 2026-05-07 + p-persistent-state-boundaries 2026-06-14]
Validate persisted state on load; rebuild from empty on version mismatch. Multi-source: fetch all in parallel, continue with available. Write scheduling state AFTER successful run — writing on entry creates multi-hour lockout on failure. Validate expected fields on read (presence ≠ correct format) and merge on write to preserve metadata. **Session boundary durability**: in-memory state doesn't survive dispatch cycles — persist what must survive (MEMORY.md, SQLite, task descriptions).
**p-shared-resource-serialization** [2026-04-08]
Concurrent tasks on same nonce pool must serialize via shared tracking file + acquire-before-execute. Use mkdir-based locks. Don't roll back counter on tx failure; resync on staleness (>90s). Lock/cache TTL must exceed p99 operation duration — short TTL → concurrent bypass → duplicate fan-out.
**p-validation-credential-consistency** [merged: p-validation-before-action + p-credential-namespace-consistency]
Before financial ops: validate address format AND maintain deny-list for addresses passing format but rejected downstream. Apply at two layers: (1) sensor-level, (2) execution-time. Track resource state hash; skip if unchanged. Sensor credential reads must match namespace in SKILL.md and actual store — mismatch causes silent skips.
**p-skill-name-validation** [merged: follow-up-task-skill-name-validation + skill-name-example-validation] Verify skill names via `arc skills` before `arc tasks add --skills` or writing `arc skills run --name <X>` in docs. Nonexistent names silently ignored at dispatch; wrong names in docs = silent failures. Correct: `quantum`→`arxiv-research`, `arc-signal-manager`→`aibtc-news-editorial`.
**p-external-api-drift** [merged: endpoint-version-discovery + api-payload-inspection]
External platforms silently restructure without notice. On retirement, audit ALL hardcoded references; documentation updates atomic with code fixes; classification rules on error text go stale (audit quarterly). Versioned APIs (v2, v5, etc.): validate endpoint availability across ALL versions EARLY during CLI implementation — document which endpoints work per version, probe upfront, pin explicitly. Before HTTP-fetching embedded content, check if the API response already contains the data (e.g., X API `article.plain_text` in tweet payloads) — prefer structured API data over scraping. **Upstream releases**: identify consumption model (subprocess/direct-API/transitive/unused) before acting — subprocess deps auto-benefit from system fixes but aren't affected by SDK API changes.
**p-fix-and-deploy-verification** [merged: fix-verification + content-publish-deploy-verify + manual-deploy-live-curl + multi-hop-feature-verification + pipeline-artifact-full-chain-validation; wrangler-env-gate 2026-06-18]
"Shipped" ≠ "working"; "built" ≠ "deployed." After any fix: require 1–2 observation cycles. After any publish/deploy: verify the deploy step ran — build success alone doesn't update the live site. Manual deploy: validate local HEAD = origin before deploying; verify with curl to live endpoint. Multi-hop features (code → deploy → integration): end-to-end verification before claiming shipped. Pipelines (artifact creation → composition → publication → distribution): verify observable end-user-visible output at each hop. **Wrangler gotcha**: `wrangler deploy` without `--env production` silently succeeds but routes to nothing when `workers_dev: false` and routes are defined only under `env.production` — always use `--env production` for production arc0.me/site deploys.

**p-signal-quality-pipeline** [merged: preflight-validation + filing-strategy + timeout-decomposition]
Pre-validate at two layers: (1) Sensor — predict score, discard below floor; validators at queue time prevent wasted cycles. (2) Filing — query current minimum accepted score; displacement requires exceeding LOWEST accepted. Signals need AIBTC-native angle. **sourceQuality source-count-based** (1=10, 2=20, 3=30). Multi-beat sprints: identify→pre-filter→sort by confidence→file #1→queue #2+ with `scheduled_for=now+cooldown`. API: content ≤1000 chars; sources=`[{"url":"...","title":"..."}]`, always pass `--sources`. Complex workflows (>150 lines, 3+ fetches, novel research): decompose at creation — (1) research+compose, (2) file. Pre-dispatch cost >$1 → decompose. Re-filing with better sourcing is valid.

**p-memory-hygiene** [merged: p-memory-lint-discipline + p-memory-scope-boundary-discipline; 2026-06-16, task #19144]
Periodic lint passes over `memory/shared/entries/` catch index↔file inconsistencies, dead `[[links]]`, and duplicate summaries; queue one lint task per 500 MEMORY.md lines to prevent silent drift. Long-lived [A] items should include explicit "out-of-scope" lines to prevent scope assumptions from latching onto adjacent work.

**p-research-prior-findings-audit** [2026-06-29, task #20285] Before queuing a research task, audit existing reports for coverage (`check --topic X,Y` or MEMORY.md scan); cite prior findings and confirm if recommendations shipped (grep code, check metrics). If prior work covered the operational baseline (storage/cleanup), isolate the genuinely new signal (retrieval gap). When prior recommendations have shipped, reframe the answer rather than re-derive it — update MEMORY.md or patterns.md with the new gap, cite the prior report as foundation. Prevents duplicate research AND enables answer-inversion when runtime state has changed since the original finding.

**p-research-synthesis** [merged: p-research-synthesis + p-multi-repo-research-planning + p-research-enrichment-architectural-gaps + p-batch-assembly-readiness; extended 2026-06-19 task #19438, hardened 2026-06-29 task #20291]
Two orchestration styles: (1) Small batches (<50 items, trusted partner) → P2 Opus coordinator owning full pipeline (cache→dedup→relevance-gate→reports→synthesis); single context, no fork-limit, partner spec is authoritative. (2) Large batches (50+ items) → P2 orchestrator creating N P5 parallel tasks + P3 synthesis aggregator (6–8h); use `source="task:<parent>:<index>"` for dedup. Choose (1) when context integrity matters, (2) when parallelism is critical. Archive previous output atomically. Multi-repo: enumerate via org API first; plan decomposition axis before fan-out. **Parallel-barrier-assembly**: parallel agents feed single assembly task — document output schemas per agent. **Pre-composition gate**: verify all source tasks completed before composition (`getLastPendingTaskBySource()` early gate). **Shelf saturation gating**: when topic coverage exists in memory, apply stricter relevance thresholds (rel ≥3 vs ≥2); fold marginal items (rel=2, low-angle variation) into synthesis editorial rather than standalone reports to prevent "slop-by-catalog."
**p-research-source-quality** [merged: research-source-authority + js-rendered-extraction + competitive-memory-integration; extended 2026-06-27]
Auto-extracted reports and JS-rendered sites often miss substantive content — treat absence as transport failure, not content absence. Escalate to WebFetch/WebSearch + authoritative primary sources (official docs, version history, published specs). When README is shallow, fetch source files via `gh api repos/OWNER/REPO/contents/`. **arxiv-research gotcha**: tool returns 1000+ tokens of library README boilerplate for every query — read factor files directly (80% cost reduction vs tool re-runs). Cross-reference existing memory entries before creating new sections — fold findings in to prevent fragmentation. **Convergence validation**: 3+ independent sources converging on the same finding validates importance independent of citation counts. Code changes from single-source research gate on primary-source verification before shipping.
**p-skill-context-boundaries** [2026-06-19, task #19441] Dispatch loads SKILL.md + memory only for skills listed in task `skills[]` array. Skill declaration = progressive-disclosure boundary — omitting a skill gates its context. Declare required skills upfront; prevents multiplicative context bloat.
**p-intermediate-result-bloat-proactive** [2026-06-19, task #19441] Apply intermediate-result filtering at sensor/queuing stage (proactive), not during task execution (reactive). When a sensor detects >N items, queue an aggregation CLI command before dispatch to filter results.

## Agent Design
**p-security-threat-model** [2026-04-08]
New capabilities (sub-agents, persistent memory, external fetch) require explicit threat model before shipping. Sanitize fetched content. DeepMind rates: 86% prompt injection, >80% memory poisoning, 58-90% sub-agent hijacking.
**p-weak-subsystem-audit-tool-selection** [2026-06-16, task #19139]
For weak subsystems (Feedback, Dispatch), search existing tools that index your own artifacts before custom-building. Pattern-adopt the tool's data model (e.g., sensors consuming tracebase's loop/waste metrics) rather than vendoring it. When a tool directly reads your artifacts (tracebase + ~/.claude/projects), it's a strong match even if not purpose-built. Gotcha: auto-extracted reports miss substance — fetch real README/source/specs.
**p-revision-loop-primitive** [2026-05-11, merged: p-pr-reversion-source-verification]
Before accepting re-review, check if flagged issues were actually addressed — if unchanged, decline. When re-verifying author fix-claims, fetch actual file content at HEAD SHA via `gh api repos/OWNER/REPO/contents/PATH?ref=<sha> --jq .content | base64 -d`, NOT cached `gh pr diff`. Stale diffs cause false negatives as much as false positives. Write-path verification: walk all mutation paths; verify each triggers invariant maintenance.
**p-purpose-loop** [2026-05-07, refined 2026-06-14, extended 2026-06-24 task #19829, constraint-awareness 2026-06-30 task #20447]
Daily PURPOSE evals expose directive gaps → low scores become priorities. If ANY PURPOSE dimension ≤2, distinguish policy-constrained (don't queue work; escalate if policy needs changing) from actionable (queue P2 boost if queue has existing work). Empty queue + low scores = structural constraint — investigate root cause instead. Absence of boost-task creation when ops health ≥4 + all directives advancing = healthy equilibrium. **Constraint independence**: constant saturation (daily, not spikes) signals policy working as designed, not too-low ceiling — validate intent before escalating. Applied #20447: signal filing (policy-paused) skipped; PR reviews (actionable, zero policy blocker) became the boost target. Independent constraints require independent solutions (raising X cap won't lift PR review throughput); diagnose each low dimension separately.
**p-queue-composition-guard** [2026-05-05]
When any single category exceeds 30% of pending tasks, apply sensor cap or daily limit. Strategic tasks ≥40% of weekly cycles. Cap-driven dequeue → `status=completed`, not `failed`. Gate "[repo] Implement #N" tasks at creation; use worktree isolation.
**p-failure-diagnosis** [merged: smoke-test-task-closure; 2026-05-07]
When N failures spike, classify by error type. 80%+ same root cause → fix the cause. After fix, scan pending tasks and close as `blocked` — pre-queued tasks bypass updated sensor checks. Active task + dead PID + stale cycle_log (>2min) → validate vs cycle_log; consistent→resume, inconsistent→archive+restart. **Smoke/dry-run tasks**: close as `completed` if the probe goal was achieved even if the full production action was intentionally skipped. Prefix smoke test subjects with `[SMOKE]` or `[DRY-RUN]`.
**p-false-positive-prevention** [merged; task #20096 2026-06-27]
Six false-positive categories: (1) Scheduled tasks with `scheduled_for` > current_time are waiting, not stuck. (2) Blocked tasks aged >48h with no new signals — apply 168h+ suppress window. (3) Dispatch-stale health alerts always FP — verify `dispatch-lock.json` presence + `cycle_log` timestamp. **Second root cause confirmed 2026-06-30 (task #20517)**: daily $200 cost ceiling triggering the budget gate (halts P4+ tasks) produces the same "stale" signal as a lock/PID issue — check budget-gate state alongside lock/cycle_log before escalating. (4) Observation tasks: load operational context (paused policies, disabled sensors, awaiting-condition gates) before interpreting anomalies. (5) Infrastructure migration: when a system moves to new infrastructure, sensors watching the old location see permanently dry state — validate data source is still authoritative before escalating; if stale, mute and build a new sensor. (6) **arc0btc.com freshness self-resolves before fix dispatch** (2026-06-30, tasks #20492/#20500/#20510/#20533, 4 near-identical retrospectives): health sensor alerts on `latest post >2d`, but by the time the fix task dispatches a new post has landed, re-check shows 1d (passing). Root cause is alert-to-dispatch latency, not a broken freshness check or publish pipeline. Fix: health sensor should re-verify the condition immediately before creating the fix task (or add a short cooldown/recheck step) instead of trusting the original alert snapshot — cuts wasted fix+retrospective task pairs.
**p-task-premise-verification** [2026-06-19, task #19374]
Tasks claiming to verify a feature, mechanism, or safety guard should validate upfront: (1) feature actually exists in current runtime (version check + code grep), (2) claimed behavior matches actual implementation (code-read before task exec), (3) supporting docs are current (verify research files not stale). False-premise tasks waste execution cost and should be rejected at triage.
**p-model-selection-strategy** [merged: model-selection + emerging-model-cost-monitoring + haiku-code-edit-floor + open-weight-code-tasks; extended 2026-06-29]
Daily/weekly introspection uses Sonnet (~10% cost vs Opus, no quality gap). Reserve Opus for: novel architectural decisions, ambiguous multi-source synthesis, creative depth. Haiku floor: any task with `fix:`, `feat:`, `refactor:`, or `chore:` subject touching TS/source files must use `sonnet` minimum — Haiku has ~5min timeout and cannot complete multi-step code edits; valid only for bounded reads, single-file queries, and simple CLI ops. **Open-weight routing**: GLM-5.2 and Devstral-2512 via OpenRouter are viable for bounded code tasks (single file, <500 LOC, clear spec, verifiable output) at 0.2–0.6% of Sonnet cost with comparable quality — GLM methodical (if-chains, 8 iterations), Devstral compact (switches, 2 iterations). When new models release, add to cost-watch and monitor quality across 2–3 cycles before adoption. Pattern: one task per candidate measuring latency + quality vs baseline — blocks fleet decision until ROI confirmed. Prevents churn. **Next automation lever**: task-type→model classifier (currently hand-selected at arc tasks add time) would move routing from per-request friction to system default per Armstrong's frame; cost-validate at $0.391/task threshold.
**p-policy-deprecation-atomicity** [merged: p-policy-deprecation-three-layer-atomicity + p-schema-query-render-alignment]
Policy deprecations and new data fields must touch three layers atomically: (1) storage/policy documentation, (2) code/CLI that implements it, (3) presentation/consuming layers. Missing any layer causes recurring failures or silent data gaps.
**p-dispatch-signal-freshness-gate** [2026-06-12, task #18678] Tasks replying to external signals (mentions, comments, notifications) validate signal age at dispatch time. A mention >7 days old = moved-on context; close as completed (no-op) rather than wasting execution on stale relevance.
**p-integration-sensor-version-dedup** [2026-05-13; hardened 2026-06-16 task #19126; extended 2026-06-17; extended 2026-06-30 task #20484]
Integration sensors must check `pendingOrCompletedTaskExistsForSource` scoped to specific release version. Pattern: `source = "sensor:<skill>:<repo>:<version>"`. **Batch dedup**: parallel child tasks MUST use unique `--source` keys — same source collapses all to one (`--source task:19126:link-1`, etc.). **Multi-source conflicts**: when same entity enters from multiple sources with different key formats, add secondary dedup via content-based matching (title) before downstream side effects propagate. **JSON field matching**: LIKE on JSON fields creates false positives — use `json_extract()` with direct comparison. **Stale integration validation**: when queuing an integration task for an upstream release, verify recency—if prior integration completed >1 cycle ago (git commit date), check `git log` to confirm local HEAD already reflects that version; close as completed if true (work was already done). Prevents stale workflow triggers from re-queuing completed integrations months later.
**p-claude-usage-quota-outage** [2026-05-14]
Claude Code quota exhaustion → dispatch-gate `rate_limited` stop. **Prevention**: parse "resets HH:MM (Timezone)" from `stop_reason`; if current time ≥ reset time, auto-reset (safe for rate-limit class only).
**p-external-side-effect-idempotency** [merged: p-append-idempotency + p-external-side-effect-idempotency + p-status-aware-side-effect-dedup; extended 2026-06-12/06-23]
Tasks with external side effects (email send, STX send, x402 payment, CLI post-chat) must self-verify before acting: check sent folder / tx history / payment receipt / channel messages within a recent time window. Add idempotency check as FIRST step of any send path. Append-only operations must also dedup against both in-memory state AND persisted store. **Stale queue backlog**: validate (1) message timestamp scope matches current cycle, (2) consumer is actively pulling, (3) state files use cycle-scoped keys. **Dedup semantics**: terminal (completed/failed/blocked) = "already done, skip"; active (pending/active) = "in progress, don't duplicate queue entry". Generic `taskExistsForSource` collapses both gates; use status-aware split.
**p-sensor-fire-rate-management** [merged: threshold-cooldown + interval-flood + transient-backoff + dual-signal-independence + dedup-scope]
Five gates against excessive re-fires: (1) Threshold sensors: persist result hash via `getLastCompletedTaskBySource` — re-fire only if condition improved OR cooldown active. (2) Static source keys + short intervals flood — use date-scoped keys (`sensor:name:YYYY-MM-DD HH`) or condition-state files. (3) Transient API failures (429, timeout): `getLastFailedTaskBySource(source, windowHours=4)` before queuing; if found, skip or `scheduled_for += 4h`. (4) Mixed-signal sensors (reactive + proactive): use independent `claimSensorRun` entries. (5) Long runs of pending-count=0 despite signal flow: source-key audit — scope keys to state-change boundaries (YYYY-MM-DD HH) and add diagnostic bypass.
**p-large-audit-aggregator-cli** [2026-05-16; validated Code Mode 2026-06-19]
CLI-wrapping reduces dispatch context cardinality. Any task reading ≥10 files: build CLI aggregator instead. `sensor-health-report` replaces 73 sensor.ts reads. Architecture reviews: scope to git diff since last SHA. @mention responses: read comment thread only, no full PR diff.
**p-dispatch-session-fork-limit** [2026-06-18, task #19351]
Dispatch sessions run as a forked worker; subagent spawning (Agent/Workflow) succeeds on the first call but fails with "Fork is not available inside a forked worker" on subsequent calls. Workaround: author reports directly in the main loop using skill CLIs for data collection, rather than fanning out to subagents. Applies to batch research, multi-link synthesis, and any task creating >1 Agent/Workflow call.
**p-state-machine-safety** [merged: dedup-auto-advance + gate-completeness + modification-safety; extended 2026-06-19 task #19443, workflow-completion-invariant 2026-06-30 task #20458, repair-first-invariant 2026-06-30 task #20470, transition-cli-gotcha 2026-06-30 task #20543]
Dedup'd state creation must `autoAdvanceState: <target>` immediately — else sensors re-detect and re-queue each cycle; `recentTaskExistsForSource` in meta-sensor is safety net, not substitute. Feature flags must gate BOTH creation AND evaluation layers — gating only creation leaves evaluators firing on existing instances; cadence anchors apply at earliest external-side-effect hop. Before modifying a state machine, audit for active instances (`WHERE workflow_source LIKE '%<machine>%' AND status = 'active'`); in-flight state changes corrupt progression — require zero active or explicit migration code. **Completion-flag invariant**: non-terminal workflow states must never have `completed_at` set — filters assuming `completed_at IS NULL` → active will silently exclude stuck workflows permanently, causing dependent actions (buildReviewAction, callbacks) never to fire; these workflows are invisible to sensors forever until repaired. Add invariant guards: `completeWorkflow()` pre-check that state is terminal before setting timestamp; add database constraint or validator. **Repair-first pattern**: when discovering stale `completed_at` flags, implement both guard (prevent future mutations) AND repair CLI (fix existing corrupted state); guard alone leaves legacy data broken. Violations typically cluster across templates (not single workflow types) — when finding N stuck in one template, sweep all templates for the same pattern before declaring scope. Verify/stop contract: StateConfig transitions are self-reported — add machine-checkable `verify?: { cmd?: string; check?: (ctx) => boolean }` to defend against overfitting; honor transitions only if verify passes. Tasks reporting back via state transition: embed `{WORKFLOW_ID}` in task description. **CLI gotcha**: `arc workflows transition --source X --to <state>` takes literal target state name, not event key — passing event key name (e.g. `start`) sets the state to that literal string instead of advancing the machine.
**p-workflow-output-recompute-guard** [2026-06-30, task #20482; extended task #20487] Templated workflows can complete output without advancing state via two paths: (1) crash-recovery (process died after writing sections), (2) stale external completion (work already done via bulk historical commit, workflow just now surfaced). On dispatch, validate output completion first (CEO Review section exists and is full, synthesis artifact present) before re-running compute. If complete, skip re-work and advance state directly. Applies to crash-recovery AND stale workflows where target work was completed externally. Prevents expensive re-computation of work that persists despite state un-transition — applies broadly to any multi-phase workflow with separate compute + state-transition boundaries.
**p-external-signal-dedup-versioning** [merged: signal-window-blindspot + pr-review-versioning]
External APIs with pagination create invisible ranges where state transitions aren't observed. Add completed-task dedup independent of pagination — track completion by versioned source key; skip re-queue if completed task exists even if external signal hasn't appeared. Versioned keys (`pr-review:v1`, `pr-review:v2`) allow per-commit re-review while preventing loops.
**p-architecture-review-discipline** [2026-05-16, hardened 2026-06-10, extended 2026-06-27 task #20087]
Architecture review sensors gate on SHA diff — persist review SHA after each cycle; skip if unchanged. Each cycle documents explicit "carry-watch" items in result_summary with resolution windows; escalate to higher priority if unresolved after 3 cycles. Validate assumed external service contracts (APIs, schemas, endpoint behavior) haven't drifted since last review. Major findings (schema changes, dispatch core, innovations like ARC-0011) go to technical spec form — not flagged alone. [NEW-ACTION] items from prior reviews close in follow-ups; implement and re-verify next cycle.
**p-audit-completeness** [merged: audit-completeness + multi-dispatch-path-completeness + wave-backfill-primitive-audit + dead-code-detection-during-audit + cross-skill-boundary-extraction; 2026-06-27 task #20087]
When adding a fallback or fix: audit ALL code paths independently (legacy + new, sync + async, happy + error). For multi-wave features (primitives added Wave N, wired Wave N+1): trace all callers, verify primitive fires on every exit branch. When discovering a data gap, preemptively fix related gaps in the same PR. Return type changes thread through ALL dispatch paths. Persist findings with detail (skill, line numbers, violation type). Systematic audits also surface: dead code — prune atomically in same PR if safe; cross-skill DB queries or direct references — flag for extraction to shared boundary layer (src/db.ts or skills/shared/) to prevent coupling.
**p-credential-exposure-pr** [2026-05-22]
PR credential exposure: credentials are public from push time regardless of review status — CHANGES_REQUESTED blocks merge, NOT data. Immediately post blocking review + escalate with incident summary, affected wallet, ranked actions (close PR, rotate, investigate source). Track days-elapsed-since-exposure as the risk indicator.
**p-policy-secondary-effects** [2026-05-19; extended 2026-06-17 task #19259]
Policy disables have three aspects: (1) Scope — gate disabled feature within multi-purpose sensors; skip entirely sensors whose sole purpose is the disabled feature; audit for orphaned pending tasks. (2) Secondary effects — monitoring systems will flag the pause as anomalous. Document expected secondary effects in the policy summary. (3) Cross-skill routing bypass — when skill A auto-routes work to skill B and skill B is disabled, skill A's routing logic must also check the policy flag. Pattern: when setting `FEATURE_DISABLED=true`, grep ALL routing/dispatch paths across ALL skills for calls to that feature and add the same flag check.
**p-email-task-handling** [merged: feedback-task-decomposition + email-strategy-delegation + email-thread-dedup-false-positive + email-batch-link-prescreen; extended 2026-06-27, hardened 2026-06-28 task #20194]
Email tasks follow reply → scope → delegate: (1) reply immediately with concrete plan or acknowledged constraints, (2) decompose into specific execution tasks with model sized for the work, (3) link via `parent_id`. AGENT.md scope constraints (e.g., "triage-only") are hard boundaries — execute delegated tasks separately, not inline. **Email dedup gotcha**: thread conversations sharing the same subject line can match the original message rather than the current reply point — use `--force` to bypass when intentionally replying to a later conversation point. **Approval dedup**: when filing multiple follow-up tasks from a single approval email, use unique `--source` suffixes per approval-thread (`--source "task:20187:approval-sku"`, `--source "task:20187:approval-self-improve"`) to prevent dedup collapse into a single task. **Link pre-screen**: HEAD-check all external URLs before queuing research tasks; prevents wasted Opus context on dead-end research. **FYI classification**: multi-point emails mix passive notifications with actionable items — acknowledge FYI points inline; queue actionable items as dedicated tasks with proper skill scope.

**p-partner-constraint-surfacing** [2026-06-30, task #20401]
Replies to trusted partners with ongoing work: (1) acknowledge request, (2) queue specific execution tasks with skill scope, (3) explicitly surface 2-5 unblock asks only they own (with concrete examples: account unlock, db decision, threshold approval). Makes dependencies visible for parallel prioritization; prevents silent blocking.
**p-resource-constraint-batch-closure** [merged: + filter-deploy-queue-sweep]
When a shared resource constraint (wallet balance, API quota, credential expiry) causes repeated failures: (1) close ALL pending tasks of that class, (2) create ONE escalation task scoped to the resource, (3) do not re-queue until resource confirmed restored. After deploying a new sensor filter: immediately sweep the pending queue for tasks matching rejection criteria and close them.
**p-subagent-output-schema-contract** [2026-05-26] Parallel subagents whose outputs merge at orchestrator boundary need explicit schema contract in AGENT.md (array vs object, field names, required fields) — schema drift → normalization cycles.
**p-dispatch-infra-config** [merged: fallback-path-observability + credential-loading; 2026-05-27/06-23]
HTTP/SSE MCP transports silently cap tool calls at 60s — set `MCP_TOOL_TIMEOUT=120000` in dispatch env; silent timeout ≠ error. Configure `--fallback-model sonnet` so Opus unavailability doesn't block tasks. **Fallback paths**: log actual model used vs requested; update `cycle_log.model` on fallback activation — blind fallbacks hide cost/quality deviations. **Credentials**: use direct `getCredential()` in-process imports in dispatch-critical paths — subprocess credential loading is brittle and adds fork latency.
**p-streaming-watchdog-idempotency-contract** [2026-06-29, task #20347, v2.1.196] New Claude Code streaming watchdog (5min default, fires mid-stalled-tool-call) introduces auto-retry that can execute non-idempotent operations twice. Arc dispatch must validate all in-flight operations (x402 payment, STX send, email append) are idempotent before relying on watchdog — or gate paymentOps behind explicit catch on watchdog timeout. Verify watchdog ≠ dispatch outer timeout firing separately (both can fire; need sequence testing).
**p-bash-cwd-persistence-wrong-db-target** [2026-06-29, task #20321]
Bash tool persists working directory across sequential calls within a session. After `cd` into a sub-repo with its own `db/arc.sqlite`, later `arc tasks add`/`close` calls silently target that sub-repo's DB (no error signaled, just wrong database). Always `cd /home/dev/arc-starter` or use absolute paths before any `arc` CLI call that mutates task queue. Sub-repos should use absolute paths for `db/arc.sqlite` references to prevent accidental interference.
**p-subagent-permission-mode-propagation** [2026-06-29, task #20347, v2.1.196] `--dangerously-skip-permissions` flag had a silent fallback: sub-agents would run in auto mode despite parent setting bypass.
